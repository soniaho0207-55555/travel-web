## 一、产品概述

### 1.1 产品定位

「环球史迹」是一款以**人类文明为核心**的移动端历史探索 H5。

区别于旅游攻略（马蜂窝）或百科（维基），本产品核心价值是：
- **惊喜感**：每次打开都有意想不到的历史发现
- **连接感**：展示文明之间的影响与关联，不是孤立的城市介绍
- **沉浸感**：全屏视觉 + 钩子文案，让用户停不下来

### 1.2 当前版本问题

| 问题 | 表现 |
|------|------|
| 内容入口太深 | 用户需多次点击才能看到实质内容 |
| 分类逻辑冷漠 | 按大洲/国家/城市的地理层级，缺乏情绪吸引力 |
| 城市数量少 | 仅 5 座城市，探索空间有限 |
| 城市孤立 | 城市之间没有任何关联性展示 |
| 首屏无钩子 | 打开即是导航列表，缺乏抓人的第一眼 |

### 1.3 目标用户

- 18–35 岁，对历史/文化/旅行有兴趣
- 碎片化时间浏览（通勤、睡前）
- 喜欢分享「涨知识」内容给朋友

### 2.1 核心体验目标

### 4.1 首页（发现页）

#### 4.1.1 今日城市 Hero

**位置**：首页顶部，全屏高度（100vh）

**视觉结构**（从上到下）：

```
┌─────────────────────────────┐
│  [Logo]          [搜索图标]  │  ← 顶部透明 Header
│                             │
│                             │
│      [ 城市实景大图 ]        │  ← 全屏背景图（Wikipedia API）
│                             │
│                             │
│  ┌─────────────────────┐   │
│  │ 🏛 帝国首都 · 3000年 │   │  ← 文明标签
│  └─────────────────────┘   │
│  北京                        │  ← 城市名（大字）
│  Beijing                    │  ← 英文名（小字）
│                             │
│  "1420年，紫禁城落成时，      │  ← 钩子文案（核心！）
│   欧洲还在黑暗的中世纪里      │
│   苦熬最后 30 年"            │
│                             │
│  [探索这座城市 →]            │  ← CTA 按钮（点击进入详情）
│                             │
│  ──────── 上滑发现更多 ──────  │  ← 引导手势提示（滚动到下方内容）
└─────────────────────────────┘
```

**交互规则**：
- 页面加载后，背景图从模糊 → 清晰（0.6s 动画）
- 钩子文案逐字出现（打字机效果，延迟 0.4s 开始）
- 点击 CTA 按钮 → 进入该城市详情页（向上滑入过渡）
- 向上滑动 → 正常滚动到下方内容（主题流、城市卡片流）
- 今日城市基于当天日期 hash 固定，不随刷新改变

**「今日」逻辑**：
```javascript
// 伪代码：基于日期选城市
const todayIndex = dayOfYear() % CITIES.length;
const todayCity = CITIES[todayIndex];
```

    flag: '🇹🇷',

## 六、导航与路由

### 6.1 页面路由规则

| 页面 | URL Hash | 说明 |
|------|----------|------|
| 首页 | `#/` | 默认入口 |
| 城市详情 | `#/city/:id` | 如 `#/city/beijing` |
| 搜索页 | `#/search` | 可携带参数 `?theme=imperial` |

### 6.2 返回逻辑

- 城市详情页 → 返回按钮 → 回到**上一页**（若从首页进入则回首页，从搜索页进入则回搜索页，保持浏览器 history 栈自然行为）
- 搜索页 → 返回按钮 → 回到首页
- 浏览器/微信返回手势 → 同上（遵循 history 栈）
- 首页滚动位置在返回时自动恢复

### 6.3 底部 Tab 切换规则

- 切换 Tab 不重置页面状态
- 首页筛选状态（选中的主题/大洲）在 Tab 切换后保留

## 七、动效规格

| 场景 | 动效 | 时长 |
|------|------|------|
| 首页 Hero 图加载 | 模糊 → 清晰（blur 20px → 0） | 600ms |
| 钩子文案出现 | 打字机逐字显示 | 40ms/字，延迟 400ms |
| 城市卡片入场 | 从下 12px + 透明度 0 → 正常 | 400ms，交错 80ms |
| 进入城市详情 | 向上滑入（translateY 100% → 0），由 CTA 按钮或城市卡片点击触发 | 350ms ease-out |
| 返回首页 | 向下滑出 | 300ms ease-in |
| 景点卡展开 | max-height 动画 | 400ms ease |
| 同年世界模块出现 | 淡入 + 上移 8px | 300ms |
| Tab 切换 | 内容区淡出淡入 | 200ms |

## 八、图片加载策略

### 8.1 城市封面图

**来源**：Wikipedia API（现有逻辑保留）

**加载时机**：
- 首页：进入首页时，批量预请求当前视口内所有城市的 wiki 图片
- 城市详情：进入详情页时请求

**占位方案**：图片未加载时显示 `heroGradient` 渐变色

### 8.2 景点图片

**来源**：Wikipedia API（现有逻辑保留）

**加载时机**：用户切换到「重要景点」Tab 时触发请求（懒加载）

**标题规范化**：使用 Wikipedia API 的 `normalized` + `redirects` 字段处理标题匹配（现有逻辑）

### 8.3 图片错误处理

```javascript
// 图片加载失败时，回退到渐变色
banner.onerror = () => {
  banner.style.backgroundImage = '';
  banner.style.background = landmark.gradient;
};
```

## 九、技术要求

### 9.1 技术栈

- **纯原生 HTML + CSS + JavaScript**（无框架依赖）
- 单文件交付（`index-h5.html`），便于直接部署到 GitHub Pages
- 不引入任何外部 JS 库

### 9.2 性能要求

| 指标 | 目标 |
|------|------|
| 首屏可交互时间（TTI） | < 2s（WiFi 环境） |
| 城市详情页切换 | < 200ms（本地渲染） |
| 图片渐进加载 | 不阻塞主线程 |
| 文件大小 | 单文件 < 300KB（不含图片） |

### 9.3 兼容性要求

| 环境 | 要求 |
|------|------|
| 微信内置浏览器 iOS | ✅ 必须支持 |
| 微信内置浏览器 Android | ✅ 必须支持 |
| Safari（iOS 14+） | ✅ 必须支持 |
| Chrome（Android） | ✅ 必须支持 |
| 桌面浏览器 | 可用即可，不做优化 |

### 9.4 适配要求

- 基准设计宽度：375px
- 最大内容宽度：480px（居中显示）
- 支持 iPhone 刘海屏（`safe-area-inset`）
- 禁用双击缩放（`user-scalable=no`）
- 文字最小 12px，点击目标最小 44×44px

### 修复总原则

在修改任何 `wiki` 字段前，**必须**使用以下 API 验证词条存在且返回正确图片：

```
https://en.wikipedia.org/w/api.php?action=query&titles=YOUR_TITLE&prop=pageimages&format=json&pithumbsize=800&origin=*&redirects=1
```

**三条铁律**：
1. 每个景点的 `wiki` 值必须**唯一**，不得与同城市其他景点或城市封面的 `wiki` 重复
2. 不得用上级/父级词条（如城市名）替代找不到图片的景点词条
3. 如 Wikipedia 确实无精准词条，优先选择**地理位置最近的代表性地标**作为替代

## 一、产品概述

### 1.1 产品定位

「环球史迹」是一款以**人类文明为核心**的移动端历史探索 H5。

区别于旅游攻略（马蜂窝）或百科（维基），本产品核心价值是：
- **惊喜感**：每次打开都有意想不到的历史发现
- **连接感**：展示文明之间的影响与关联，不是孤立的城市介绍
- **沉浸感**：全屏视觉 + 钩子文案，让用户停不下来

### 1.2 当前版本问题

| 问题 | 表现 |
|------|------|
| 内容入口太深 | 用户需多次点击才能看到实质内容 |
| 分类逻辑冷漠 | 按大洲/国家/城市的地理层级，缺乏情绪吸引力 |
| 城市数量少 | 仅 5 座城市，探索空间有限 |
| 城市孤立 | 城市之间没有任何关联性展示 |
| 首屏无钩子 | 打开即是导航列表，缺乏抓人的第一眼 |

### 1.3 目标用户

- 18–35 岁，对历史/文化/旅行有兴趣
- 碎片化时间浏览（通勤、睡前）
- 喜欢分享「涨知识」内容给朋友

## 二、产品目标

### 2.1 核心体验目标

> 用户打开页面的前 10 秒就被震撼到，愿意继续探索。

### 2.2 量化指标（供参考）

| 指标 | 目标 |
|------|------|

| 人均浏览城市数 | > 3 座/次 |

#### 4.1.1 今日城市 Hero

**位置**：首页顶部，全屏高度（100vh）

**视觉结构**（从上到下）：

```

│                             │
│                             │
│      [ 城市实景大图 ]        │  ← 全屏背景图（Wikipedia API）
│                             │
│                             │
│  ┌─────────────────────┐   │
│  │ [icon] 帝国首都 · 3000年 │ ← 文明标签（SVG，非 emoji）
│  └─────────────────────┘   │
│  北京                        │  ← 城市名（大字）
│  Beijing                    │  ← 英文名（小字）
│                             │
│  "1420年，紫禁城落成时，      │  ← 钩子文案（核心！）
│   欧洲还在黑暗的中世纪里      │
│   苦熬最后 30 年"            │
│                             │
│  [探索这座城市 →]            │  ← CTA 按钮（点击进入详情）
│                             │
│              ↓              │  ← 细线 SVG 下箭头（呼吸动效，非文字）
└─────────────────────────────┘
```

**交互规则**：
- 页面加载后，背景图从 `blur(10px)` → 清晰，**同步 opacity 0 → 1**（600ms，胶卷显影感）
- 钩子文案逐字出现（打字机效果，延迟 0.4s 开始）
- 点击 Hero 区域任意位置（城市名、图片、CTA 按钮）均可进入该城市详情页（向上滑入过渡）
- 向上滑动 → 正常滚动到下方内容（主题流、城市卡片流）
- 今日城市基于当天日期 hash 固定，不随刷新改变

> **v2.2 变更**：
> - Hero blur 从 20px 降到 10px（避免"用力过猛的失焦感"）
> - 原"—— 上滑发现更多 ——"文字提示 → **细线 SVG 下箭头 + 呼吸动效**（UI 语言自己说话，无需文字）
> - 文明标签的 emoji（🏛/👑 等）→ 金色 monoline SVG（见 9.5.1 图标替换清单）

**「今日」逻辑**：
```javascript
// 伪代码：基于日期选城市
const todayIndex = dayOfYear() % CITIES.length;
const todayCity = CITIES[todayIndex];
```

##### 4.2.3·a 【v2.3 重构】景点信息模块（门票 4 件套 + Tips 数组）

> **v2.3 覆盖 v2.2 字段**：v2.2 定义的扁平字段 `ticketUrl` / `advanceBooking` / `bestTime` 本版**全部作废**，未上线前直接换成下述结构化对象。v2.2 字段名**不进 js/data.js**，Dev 请按 v2.3 字段实现。

**数据模型**（`js/data.js` 的 `landmarks[]`）：

| `secret`  | 隐藏彩蛋 | key-line       | "地下层 Hypogeum 走角斗士通道" |

**Tips 内容标准**（写入 PRD，Dev/PM/Content 三方共同约束）：

> 每条 tip 必须命中以下**至少一项具体锚点**。一项都不命中 = 该条由 PM 重写，**不计入 ≥4 条的合格数量**（而非 FAIL 整个 PR）。

| 锚点类型 | 举例 |
|----------|------|
| **数字** | "联票 48h 有效"、"珍宝馆加 ¥10"、"下午 4 点天黑" |
| **具体时间** | "开门前 8:30 到"、"落日前 1 小时最美" |
| **具体方向/位置** | "康斯坦丁凯旋门方向拍西立面"、"神武门外向上拍景山" |
| **具体名字** | "地下层 Hypogeum"、"珍妃井旁澡堂遗址" |
| **只有去过的人才知道的细节** | "中轴线走完再去东六宫，人比西六宫少 40%" |

**反面例子**（都不命中 → 不合格）：
- "建议网上预约"（无数字、无具体时间）
- "请穿舒适鞋"（无任何锚点）
- "注意防晒"、"带好证件"（纯常识）
- "避开人多时段"（无具体时间）

**好例子**（任一锚点即合格）：
- "11-14 点团客最密，避开"（具体时间 ✓）
- "地下层 Hypogeum 走角斗士通道，90% 游客不知道"（具体名字 + 去过才知道 ✓）

**空状态处理**：

- `ticket.channels` 空数组 → 不渲染"购票渠道"块
- `ticket.bookingWindow` 字段缺失或三字段全空 → 不渲染"预约提前"块
- `ticket.timingTip` 空 → 不渲染"时段建议"块
- `tips` 空数组 → 不渲染 Tips 模块（不显示"暂无"占位）
- `ticket.price` 必填（沿用 v2.0 要求）

#### 4.2.4 【v2.1 新增】时间轴世界背景注释（`worldContext`）

**背景**：UX 反馈指出时间轴只有本城市事件，缺乏世界纵深感；而景点模块的"同年世界"对比叙事受到好评。

**设计原则**：景点模块的 `worldEvents` 通过"展开景点"触发，是惊喜呈现。时间轴若每个节点都加重度对比，会稀释惊喜感。因此时间轴采用**轻量级**的单句时代背景，仅覆盖关键节点。

**触发位置**：时间轴节点的 `desc` 文字下方，灰色小字注释（非独立卡片）。

**展示逻辑**：仅在 `timeline` 事件显式包含 `worldContext` 字段时显示，其他节点不显示。

**视觉结构**：
```
┌─────────────────────────────────────┐
│  ● 1420 年                          │
│    永乐迁都·紫禁城破土                │  ← 事件（现有）
│    明成祖朱棣动员逾百万工匠…         │  ← desc（现有）
│    ─────────────                    │
│    🌍 此时世界：欧洲文艺复兴正萌芽，   │  ← worldContext（新增）
│       奥斯曼帝国正积蓄力量            │     灰色小字、左侧金色竖线
└─────────────────────────────────────┘
```

**数据结构**（时间轴节点新增可选字段）：
```javascript
timeline: [
  {
    year: '1406年',
    event: '永乐迁都·紫禁城破土',
    desc: '明成祖朱棣动员逾百万工匠，历时十四年建成紫禁城……',
    worldContext: '此时欧洲文艺复兴正萌芽，奥斯曼帝国积蓄力量准备攻陷君士坦丁堡'  // 【新增·可选】
  }
]
```

**内容编辑规范**（PM 负责撰写）：
- **覆盖范围**：每座城市的时间轴挑选 **2–3 个关键转折节点** 补充，不是每个节点都写
- **长度**：单句，不超过 50 字
- **以"此时世界/此时全球/彼时 XX"开头**，与景点 worldEvents 的"同一年 XX 正在 YY"句式区分
- **内容**：以宏观文明对比为主，不堆砌具体人名/数字
- **禁止**：国旗 emoji + 国家列表的格式（避免和景点 worldEvents 视觉混淆）

**空状态**：若时间轴节点无 `worldContext` 字段，则不显示该行，不留空白占位。

**验收标准**：
- [ ] 15 座城市每城至少 2 个时间轴节点含 `worldContext`
- [ ] 展示形式与景点 worldEvents 视觉区分明显（单句 vs 多国卡片）
- [ ] 无 `worldContext` 的节点不影响原有时间轴渲染

## 六、导航与路由

### 6.1 页面路由规则

| 页面 | URL Hash | 说明 |
|------|----------|------|
| 首页 | `#/` | 默认入口 |
| 城市详情 | `#/city/:id` | 如 `#/city/beijing` |
| 搜索页 | `#/search` | 可携带参数 `?theme=imperial` |

### 6.2 无效路由处理

- 访问不存在的城市 ID（如 `#/city/cusco`）→ 显示提示"未找到该城市"，并推荐 3 座热门城市供用户选择
- 访问未定义的 hash 路由 → 重定向到首页

### 6.3 返回逻辑

- 城市详情页 → 返回按钮 → 回到**上一页**（若从首页进入则回首页，从搜索页进入则回搜索页，保持浏览器 history 栈自然行为）
- 搜索页 → 返回按钮 → 回到首页
- 浏览器/微信返回手势 → 同上（遵循 history 栈）
- 首页滚动位置在返回时自动恢复

### 6.4 底部 Tab 切换规则

- 切换 Tab 不重置页面状态
- 首页筛选状态（选中的主题/大洲）在 Tab 切换后保留

### 6.5 底部导航在城市详情页的行为

- 城市详情页**隐藏底部导航**，给内容更多空间
- 或：保留底部导航但不高亮任何 Tab，用户仍可点击"发现"/"搜索"返回对应页面

### C-01 🔴 P0 | 全站图片 background-size 被覆盖导致模糊

**现象**：首页城市卡片、景点 banner、搜索缩略图大量显示为模糊渐变色块，图片实际已加载成功但未正确显示。

**根因**：HTML 中用 `style="background: ${gradient}"` 设置初始渐变，`background` 简写属性会将 `background-size` 重置为 `auto`。后续 JS 用 `el.style.backgroundImage = url(...)` 替换图片时，`background-size` 仍为 `auto`，CSS 中的 `background-size: cover` 被内联样式覆盖。

**影响范围**：
- `app.js:163` — 城市卡片 `.city-card-img`
- `app.js:266` — 景点 banner `.landmark-banner`
- `app.js:445` — 搜索结果缩略图 `.search-result-thumb`

**修复方向**（二选一）：
1. 将所有 `style="background: ${gradient}"` 改为 `style="background-image: ${gradient}"`，保留 CSS 中的 `background-size: cover`
2. 在 JS 赋值图片 URL 时，同时设置 `el.style.backgroundSize = 'cover'`

**验收标准**：
- [ ] 所有城市卡片图片清晰显示为 cover 模式
- [ ] 所有景点 banner 图片清晰显示
- [ ] 搜索结果缩略图清晰显示
- [ ] Wikipedia API 缓存中的图片 URL 均能正确渲染

### E-01 🔴 P0 | 景点门票信息 4 件套（数据模型重构）

**对应 UX 反馈**：C-3（门票购买 & 预约 tips 完全缺失）

**现象**：当前 `ticket` 只是 "€18 · 09:00-19:00 · 建议网上预约" 三行——没官方 URL、没渠道对比、没淡旺季窗口、没时段建议——用户看完会退出 App 跳转马蜂窝，转化链条断掉。

**需求**：`js/data.js` 中所有 `landmarks[].ticket` 从 string 改为 object，结构见 PRD 4.2.3·a。

```javascript
ticket: {
  price: string,                              // 必填
  channels: [{ name, url, note }],            // 可选数组（空则不渲染渠道块）
  bookingWindow: {                            // 可选对象
    peak: string,        // 旺季
    shoulder: string,    // 肩季
    offpeak: string      // 淡季
  },
  timingTip: string                           // 可选
}
```

**UI 分 4 子块**：票价 / 购票渠道 / 预约提前 / 时段建议 — 每块独立可空，空则不渲染、不留占位。

**数据研究源**（PM / Content 按优先级）：马蜂窝 → Tripadvisor → Reddit r/travel → 小红书。

**验收标准**：
- [ ] 15 座城市的 50+ 景点 `ticket` 字段全量迁移为 object 结构
- [ ] 4 个子块空状态独立处理，任一空不报错
- [ ] 购票渠道 `url` 在新 tab 打开（`target="_blank" rel="noopener"`）
- [ ] 渠道行显示外链图标（右上箭头 SVG）
- [ ] 首批 5 个景点（斗兽场 / 故宫 / 圣索菲亚 / 金字塔 / 马丘比丘）必须补齐全部 4 子块样本

### E-02 🔴 P0 | 景点 Tips 数组重构（分类 + 7 SVG 图标 + 锚点清单）

**对应 UX 反馈**：C-4（每个景点 Tips 只有一条 · 全是常识）

**现象**：当前每景点单字段 `note`，且内容是"建议网上预约"这种常识——"一条常识 tip = 不如没有"。不做主观否定，采用 PRD 4.2.3·a 的**锚点清单**做客观正向判定。

**需求**：新增 `landmarks[].tips` 数组字段，见 PRD 4.2.3·a。

```javascript
tips: [
  { category: 'ticket'|'timing'|'photo'|'route'|'dress'|'season'|'secret', text: string },
  // ≥4 条，含至少 3 种不同 category
]
```

**7 个新 SVG 图标**（入库，见 9.5.1 扩充）：

`ticket-dashed` / `clock-thin` / `camera-line` / `footprint-line` / `shirt-line` / `snowflake-line` / `key-line`

**UI 样式**：米色半透明 list（杂志 Editor's Note 感），每条一行 ≤30 字，左侧分类 SVG 16×16 哑金。

**Tips 内容标准**（详见 PRD 4.2.3·a 锚点清单）：
- 每条 tip 必须命中至少一项具体锚点：**数字 / 具体时间 / 具体方向 / 具体名字 / 只有去过才知道的细节**
- 未命中的 tip 由 PM 重写，**不计入 ≥4 条的合格数量**（而非 FAIL 整个 PR）
- Dev/PM 提交前对照锚点清单自检，降低来回打回概率

**PM 承诺交付**：首批 3 个景点（斗兽场 / 故宫 / 圣索菲亚）各 6 条 tips 做样本（共 18 条），其余景点 Dev 先留 `tips: []`，PM 按城市分批补齐。

**验收标准**：
- [ ] `tips` 字段类型定义正确，空数组不报错不渲染
- [ ] 7 个分类 SVG 图标入库并按 `category` 自动映射
- [ ] 首批 3 个景点各 ≥4 条 tips，含 ≥3 种不同 category
- [ ] QA 抽检 5 条 tips 对照 PRD 4.2.3·a 锚点清单，未命中的标红退回 PM（不计入 ≥4 条合格数）

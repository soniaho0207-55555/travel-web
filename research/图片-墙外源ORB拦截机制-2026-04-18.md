# research: 图片 - 墙外源 ORB 拦截机制 - 2026-04-18

## 背景

travel-web 多轮 UX 反馈提到：京都、吴哥窟等城市 Hero 图"黑屏"/"不加载"，控制台偶尔看到 `net::ERR_BLOCKED_BY_ORB`。CEO 和 UX 都怀疑是**浏览器 ORB（Opaque Response Blocking）机制**在作祟。

本研究的任务：**把 ORB 这个黑盒打开**——它到底是什么、什么时候触发、是不是 travel-web 黑屏的真正原因。

**研究中最重要的诚实**：做完资料消化后，我发现 **travel-web 的图片黑屏极大概率不是 ORB 造成的**，而是**中国大陆 GFW + hotlink + adblock 三层夹击**。ORB 只是浏览器控制台显示的错误文字，不是根因。本研究的主要价值是**帮 UX 以后能看懂控制台报错，准确归因**。

## 1. ORB 是什么——先祛魅

ORB（Opaque Response Blocking）是 Chrome 从 2022 年开始部署的一种**浏览器侧安全机制**，前身叫 CORB（Cross-Origin Read Blocking）。官方称号 "CORB++"。

### 1.1 ORB 解决的是什么问题

**不是解决"盗链"**，不是解决"墙"，是解决**侧信道攻击（side-channel attack）**：

恶意网站 A 可以通过 `<img src="https://bank.com/account-data.json">` 这种方式**加载本不该被当作图片的敏感数据**，然后通过图片尺寸、加载时间等侧信道推测内容。ORB 就是在浏览器层**检测这种"mime 类型撒谎"**，阻止非图片内容被当作图片加载。

### 1.2 ORB vs CORB 的差别

| | CORB（旧） | ORB（新） |
|---|---|---|
| 阻止时做什么 | 返回空响应（静默）| 抛网络错误（loud）|
| 检测范围 | 部分 MIME 类型 | 全量非安全列表 |
| 错误显示 | 用户看不到 | `net::ERR_BLOCKED_BY_ORB` 控制台 |

换句话说，**ORB 更"吵"，用户更容易注意到**——这就是 travel-web 控制台看到这个错误的来源。

## 2. ORB 的触发算法（来自 annevk/orb 规范）

ORB 判断一个跨源响应是否 block，按以下 5 步：

### Step 1 · 提取 MIME 类型 + 检查 nosniff

读 `Content-Type` header 和 `X-Content-Type-Options: nosniff` header。

### Step 2 · 安全列表（直接放行）

以下 MIME 类型**直接允许通过**（ORB 不过问）：
- JavaScript (`text/javascript`, `application/javascript` 等)
- CSS (`text/css`)
- 各种图片类型（`image/jpeg`, `image/png`, `image/webp`, `image/avif`, `image/svg+xml` 等）
- 媒体（音视频）

**关键结论**：**只要服务器返回了正确的 `Content-Type: image/*`，图片永远不会被 ORB 拦截**。

### Step 3 · 阻止列表（直接 block）

以下 MIME 立刻 block：
- HTML (`text/html`)
- JSON (`application/json`)
- XML
- PDF、压缩包、Office 文档

### Step 4 · 内容嗅探（MIME 不明确时）

如果 `Content-Type` 是 `text/plain` 或缺失，浏览器会读前 1024 字节，做 MIME sniffing：
- 匹配图片 magic bytes（JPEG FFD8FFE0, PNG 89504E47, WebP RIFF…）→ 放行
- 匹配不到 → block

### Step 5 · JS 解析验证

如果嗅探仍不明确，会尝试作为 JavaScript 解析——能解析且不是 JSON，放行。

## 3. 那图片为什么会被 ORB 拦截？

根据规范，**合规服务器返回的图片根本不应该被 ORB 拦截**。如果被拦截了，**一定是服务器的错**：

| 服务器错误 | 症状 | 修复 |
|---|---|---|
| Content-Type 返回 `text/html` 或 `text/plain` | ORB 看到不是图片类型 → 走嗅探 → 可能 block | 服务器必须返回 `image/jpeg` 等正确 MIME |
| 返回错误页（404/403）带 HTML 内容 | 内容是 HTML 即使 URL 看起来像图片，也被 block | 返回错误时 Content-Type 也要匹配，或者前端用 `onerror` 兜底 |
| 缺失 `X-Content-Type-Options: nosniff` 且内容混杂 | 浏览器可能不信任服务器的 Content-Type | 加 `X-Content-Type-Options: nosniff` header |

### 3.1 travel-web 真正的黑屏成因（推测）

对应到 travel-web：
- **京都/吴哥窟 Hero 图用的是 Google Photos / googleusercontent.com 的链接**
- 这个 host 在中国大陆**被 GFW 网络层屏蔽**（DNS 污染 + IP 封锁）
- 浏览器发起请求 → 连不上 → **超时 → 返回空响应或错误**
- 此时 ORB 看到的不是合法图片响应 → 走嗅探 → **嗅探失败 → `net::ERR_BLOCKED_BY_ORB`**

**所以 ORB 只是"最后一个显示的错误"，根因是网络层连不上**。

## 4. 中国大陆场景 — 真正的拦截来自多层

travel-web 在中国大陆用户那里看到的图片问题，实际是**四层拦截的叠加**：

### 层 1 · GFW 网络层（最主要）

以下 host **在中国大陆大概率被墙或不稳定**：
- `googleusercontent.com`、`lh3/4/5/6.googleusercontent.com` → Google Photos 图床
- `gstatic.com`、`googleapis.com` → Google 静态资源
- `imgur.com`、`i.imgur.com` → Imgur 图床
- `unsplash.com`（部分线路可达但慢）
- `i.redd.it` → Reddit
- `cdn.jsdelivr.net`（部分节点被墙，不稳定）

**错误表现**：`net::ERR_CONNECTION_TIMED_OUT` / `net::ERR_CONNECTION_REFUSED` / 或者等很久后 `net::ERR_BLOCKED_BY_ORB`（因为连接失败后空响应被 ORB 二次判定）。

### 层 2 · 图床的 hotlink 保护

一些图床有 referrer 检查，站外加载时会拒绝：
- Imgur 对非 imgur.com 站点的 referrer 可能返回 403
- 小红书图床（`ci.xiaohongshu.com`）可能 refer 检查
- 部分微博图床

**错误表现**：HTTP 403/401 或返回占位错误图。

### 层 3 · ORB（浏览器侧）

如前所述，ORB 本身**只在服务器 MIME 类型不合规或内容有问题时**拦截。是**所有请求错误的"末端显示"**，不是根因。

### 层 4 · 用户侧 adblock 插件

**错误表现**：`net::ERR_BLOCKED_BY_CLIENT`（注意这个和 ORB 不是同一个）。

## 5. UX 应该怎么看控制台错误（快速归因表）

下次 UX 上线体验遇到图片问题，打开 DevTools → Network → Console，按下表快速归因：

| 控制台错误 / Network 状态 | 真实原因 | 修复责任方 |
|---|---|---|
| `net::ERR_CONNECTION_TIMED_OUT` on `googleusercontent.com` | **GFW 网络层拦截** | Dev-H5（迁 CDN）|
| `net::ERR_NAME_NOT_RESOLVED` | DNS 污染或 host 本身下线 | Dev-H5（换 host）|
| HTTP 403/401 on `imgur.com` 等 | Hotlink 保护 | Dev-H5（换 host 或代理）|
| `net::ERR_BLOCKED_BY_CLIENT` | **用户 adblock 误伤** | 不可控（但可换 host 规避 fingerprint）|
| `net::ERR_BLOCKED_BY_ORB` 单独出现 | 服务器 Content-Type 错误（罕见） | Dev-H5（修服务器 header 或换 host）|
| HTTP 200 但图不显示、控制台无报错 | `<img>` 用了错误的 crossorigin 属性 | Dev-H5（检查前端代码）|
| Network tab status "pending" 长时间不变 | GFW 或服务器卡死 | Dev-H5（换 host）|

**核心归因口诀**：
- ERR_CONNECTION / ERR_NAME → 网络/GFW 层（95% 是墙）
- ERR_BLOCKED_BY_CLIENT → 用户 adblock
- ERR_BLOCKED_BY_ORB → 服务器 MIME 问题（罕见）
- HTTP 403/401 → hotlink 保护

## 6. 修复路径（按优先级）

### 6.1 终极方案 · 全部图片迁国内 CDN（**P0，强推**）

即 Plan B 的 B.1 研究结论：选腾讯 COS / 火山 ImageX 之一，绑定自定义域名（`img.travelweb.com`），把所有墙外 host 的图片 migrate 过去。

**一次性解决**：GFW、hotlink、ORB、adblock 误伤——全部消失。

### 6.2 过渡方案 · Cloudflare 国内优化或自建反代（P1）

如果短期内不能迁 CDN，可以：
- 用 Cloudflare（选中国优化线路）作为图片代理，把 `lh3.googleusercontent.com/xxx` 重写成 `img.travelweb.com/proxy/lh3/xxx`
- 自建 Nginx 反代（需要海外服务器 + 国内 CDN 回源）

### 6.3 兜底方案 · `<img>` onerror 占位图（**保底，但治标**）

前端给所有 `<img>` 加 `onerror` 处理器，加载失败时替换为占位图：

```html
<img src="https://lh3.googleusercontent.com/xxx" 
     onerror="this.src='/fallback.jpg'"
     alt="京都">
```

或者：

```javascript
// 监听所有图片加载失败
document.addEventListener('error', (e) => {
  if (e.target.tagName === 'IMG') {
    e.target.src = '/fallback.jpg';
  }
}, true);
```

**治标不治本**：用户还是看到的是占位图而不是原图，体验割裂。

### 6.4 警告：不要乱用 `<meta referrerpolicy>` 或 `crossorigin` 属性

这两个属性**不能解决 GFW 拦截问题**，但有些开发者会试图用它们来"fix ORB"——结果只会掩盖错误（ORB 错误消失了但图还是加载不出来）。不要走这条歪路。

## 7. 对 travel-web 的具体行动建议

### 7.1 立刻（本周内）

1. **让 UX 下次上线时，对每个加载失败的图片记录控制台错误类型**（用上面 §5 的归因表）
2. **全站扫描一遍 `js/data.js`，列出所有外部图片域名**，标记哪些在大陆大概率挂（`googleusercontent`、`imgur`、`gstatic` 等）
3. **加 onerror 占位**作为立即上线的保底（1-2 小时工作量）

### 7.2 短期（2-4 周）

4. **执行 Plan B B.1 的结论**：开通腾讯 COS 或火山 ImageX，绑定自定义域名
5. **迁移所有城市 Hero 图 + 景点图到新 CDN**（同时做 WebP 转换）
6. **更新 js/data.js 里的所有图片 URL**

### 7.3 长期（1-3 个月）

7. **加一层自建 image server**（Cloudflare Workers / Next.js Image），让 URL 变成 `img.travelweb.com/kyoto/hero?w=800` 这种干净格式
8. **建立图片 quality gate**：新加的城市必须用自家 CDN，禁止墙外 host

## 9. 局限与警示

- **ORB 算法时效性**：Chrome 的 ORB 还在 v0.2，未来可能调整。本研究的结论短期（1-2 年）有效，长期要跟进 chromestatus。
- **没做实地抓包**：理论分析为主。travel-web 的真实黑屏需要 UX 用 DevTools 抓一次真实错误才能 100% 确认是 GFW 还是别的。建议 UX 下轮必做。
- **GFW 是动态的**：具体哪个 host 在大陆能通，每月都可能变。`googleusercontent.com` 现在稳定被墙，但 `unsplash.com` 有些线路能通、有些不能——这不是确定性问题。
- **iOS Safari / 微信 WebView 的 ORB 实现不完全同 Chrome**：本研究基于 Chromium。微信内置 WebView（X5 / TBS）有自己的处理逻辑，可能和 Chrome 行为不同——需要单独测试。
- **单靠前端修不了 GFW 问题**：这个结论必须反复强调。任何"加个 crossorigin 属性" / "用 fetch 替代 img" 的建议都是**治标**——真正的解是迁 CDN。

## 10. 本研究的一个元结论（给 CEO 和 research-ux 自己）

这是第一个"我做完以后发现原假设是错的"的研究：

**出发假设**："ORB 是罪魁祸首，要研究 ORB 怎么办"
**研究结论**："ORB 不是根因，GFW 才是。ORB 只是错误显示的最后一环"

这提醒我：UX 吐槽的控制台错误 ≠ 真实问题。research-ux 的价值之一是**从错误表象追到真正的系统原因**，不能照着错误信息表面修。

镜头 1（"图片错误归因训练"）就是这个元结论的镜头化——让 UX 和 Dev 以后都知道"错误信息本身可能在撒谎"。

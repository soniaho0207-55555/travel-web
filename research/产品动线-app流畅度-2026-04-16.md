# research: 产品动线 - app 流畅度（iOS 相册 / Airbnb / Apple Music / 小红书） - 2026-04-16

## 背景

上一次「流畅度 - Linear」研究聚焦在**单次操作的即时反馈**（tap 响应、loading 形态、scroll 保持）。但 travel-web 真正的痛点是**多步骤路径上的连贯感**——城市卡片 → 详情页 → 返回列表 → 看下一个城市，这条动线要让人觉得"在逛"，不是"在跳页"。

这次研究换个角度：**「浏览 → 详情 → 返回」这条同构动线**，看四个标杆 app 各自怎么解。它们的共性不是"动效更花"，而是**对动线断点的强迫症**：每一处可能让用户感到"被打断"的地方都被打磨过。

## 核心发现：动线流畅 = 把"断点"全消掉

### 1. 共享元素过渡（Shared Element / Hero Animation）

**iOS 相册经典样本**：缩略图网格 → 点一张 → **那张图本身**长大、移动、占满屏幕。视觉上"图没换"，只是位置和尺寸变了。详情页就是用户点的那张图自然成长出来的。

**Airbnb 同样手法**：Google Design 案例直接说「photographs animating and enlarging on tap to anchor each listing」。一个 listing 卡片的房屋照片，点开后就是详情页第一屏的大图——同一张图在两个页面之间"流"过去。

**为什么这件事重要**：因为人脑在处理"位置变化"时比处理"切换页面"流畅 10 倍。共享元素让大脑相信"这是同一个东西在动"，于是没有"换了一页"的感受——所以根本没有断点。

**反例（绝大多数 web 站）**：列表用一张缩略图，详情用另一张大图，中间用路由切换 = 用户视觉上经历"图消失 → 白屏 → 新图出现" = 三个断点。哪怕加 fade transition，也只是把断点伪装得不那么明显。

### 2. 过渡的方向一致性（Directionality）

iOS 26 motion guide：「clear and consistent direction rules, so users can easily follow what's happening on screen」。

**iOS 原生骨架**：
- push（进入下一层）= 新页面从右滑入，旧页面同时左移
- pop（返回上一层）= 旧页面从左滑回，新页面右移退出
- modal（弹出独立层）= 从下往上推
- dismiss modal = 从上往下落

这套方向语法让用户**永远知道"我现在在哪一层"** —— 滑入是更深，滑出是回退，弹起是临时层。

**反例**：Web 上很多站点进入和返回都是 `fade-in` —— 用户失去"我去了哪里、我从哪里回来"的空间感。短期觉得"挺干净"，长期觉得"找不着北"。

### 3. 过渡时长甜点（100-500ms 区间）

iOS 26 guide 给的具体范围：
- 快速交互（如按钮反馈）：「usually under 200ms」
- 页面过渡：「about 300–500ms」
- 总体：「between 100ms and 500ms」

**为什么这是甜点**：
- < 100ms：用户的眼睛跟不上，等同于"瞬间换"，反而像 bug
- 100-300ms：感知到"正在过渡"但没等待感
- 300-500ms：明显感知，但仍在容忍区
- > 500ms：开始觉得"卡"

**Apple 没说但行业共识**：缓动曲线（easing）比时长更影响"感觉"——`ease-out`（前快后慢）让东西"落"在终点，比线性 / `ease-in` 更接近物理直觉。CSS 默认的 `ease`（`cubic-bezier(0.25, 0.1, 0.25, 1)`）对绝大多数过渡是合格的；想更"Apple 感"用 `cubic-bezier(0.4, 0, 0.2, 1)`（Material 也是这条）。

### 4. 持续在场（Persistent UI）—— Apple Music mini player 模式

Apple Music：「播放器是音乐应用的心脏，应该始终在手边」。

**机制**：用户在浏览专辑、看歌单、找艺人时，正在播放的歌**不消失**——底部一条 mini player 横条永远在场，左侧专辑封面、中间歌名、右侧播放/暂停。

**为什么这条对动线流畅至关重要**：因为它取消了"切回去看现在播什么"这个动作——这个动作每次出现，都是动线断裂。Apple Music 把"现在播什么"做成持续在场的常量，从而让"浏览"和"播放"两条线**永远不打架**。

**对 travel-web 的迁移想象**：城市详情页里，"我从哪个列表来"是不是有持续提示？面包屑是一种弱形态，更强的是"返回上一城市列表 (12 城)"这样具体的"现场"指针。

### 5. 连续浏览不被打断（Continuous Browsing）

小红书 RCP 框架的存在原因：「滑动场景占小红书用户体验的 90% 到 95%」。一旦在双列瀑布流滑动中突然出现 spinner / 全屏 loading，"刷"的节奏立刻被切——用户的注意力从"看下一张"转到"等"。

**小红书的解法**：预加载（用户还在看第 5 张时，第 8-10 张已经在请求）+ skeleton（占位先出来）+ 极端情况下 silent loading（不出 spinner）。

**核心原则**：**连续浏览中的任何 spinner 都是失败**。Loading 必须出现在动线"自然停顿点"（如初次进入页面、主动操作触发）—— 在用户**正在滑动**时绝对不能弹出。

### 6. 返回路径对称（Symmetric Entry/Exit）

Apple Music 在 Mac 版被吐槽：「从全窗口切到迷你播放器很容易，反向很复杂，必须进入'窗口'菜单再找迷你播放器」。

**这是一个被广泛犯的设计错误**：进入路径被精心设计（首页 → 引导 → 模态 → 完成），退出路径被遗忘（用户得自己找返回 / 关闭按钮）。

**对称原则**：用户用什么动作进，就该用对称的反向动作退出——
- 点击进 → 点击退（X 按钮、返回按钮要在显眼位置）
- 滑入进 → 反向滑出（iOS 边缘左滑返回 = 视觉对称）
- 弹起进 → 下拉关闭（modal 标准）

不对称 = 用户进得去出不来 = 动线被堵 = 流畅度断裂。

## 不值得学 / 反而要避开的

### 1. 把动效当装饰

Airbnb / iOS / 小红书的动效都**功能性**——共享元素是为了视觉连续，方向一致是为了空间感，时长甜点是为了感知节奏。**反例**是"加动效是为了好看"——弹簧晃动、3D flip、视差摄影机都是"看我"型动画，违反 Linear 那条「structure should be felt not seen」。

### 2. 复刻原生 app 的所有动线（H5 装 native）

travel-web 是 H5，**不可能也不必须**做到 100% 原生流畅度。该用 web 的方式表达"流畅"：
- CSS view-transition API（Chrome/Safari 现代版本支持）
- CSS scroll-snap（iOS Safari 已支持）
- prefers-reduced-motion 媒体查询（无障碍）

而**不是**硬上 React Native / 原生 webview 套壳来"装"原生。装得不像反而像 H5。

### 3. 移植小红书"刷感"

小红书的"刷"是商业模式的果，不是手段——他们要用户停留长，所以刷感被极致优化。travel-web 是内容站不是消费时间产品。**别照抄无限滑动**——城市内容站需要"看了 3 个城市觉得满足"，不是"刷不停"。

## 可迁移到 travel-web 的具体灵感

### travel-web 当前动线大概率有的 5 个断点

1. **列表 → 详情是路由跳转 + 白屏**（最严重）
   → 改成 view-transition + shared element：城市卡片图变成详情页第一屏图，整个过程 300ms

2. **进入和返回都是 fade（无方向感）**
   → 进入用 slide-in-right，返回用 slide-out-right（CSS view-transition 自动处理）

3. **过渡时长不是 0 就是 1s**
   → 全站统一到 200-350ms 区间，缓动用 `cubic-bezier(0.4, 0, 0.2, 1)`

4. **连续滑动中弹出 loading**
   → 图片用 `loading="lazy"` + 预占位（aspect-ratio 早出）；详情数据预取

5. **详情页"我从哪来"无提示**
   → 返回按钮要明显（不是浏览器自带），最好带文字"返回 12 城列表"

### 不需要新功能也能做的 3 件

- 加 CSS view-transition-name（一行属性绑定列表卡片图和详情图）
- 替换路由的 fade 为 slide（Vue Router / React Router 都有内置）
- 全站统一 transition timing 变量（CSS custom property）

## 一句话总结

**动线流畅 = 把所有"用户能感觉到换页"的瞬间消掉。** Cereal 教会内容克制，Linear 教会单点反馈，这次研究教会的是：**两个屏幕之间的那 300ms，决定了用户觉得"在逛"还是"在查"**。共享元素 + 方向一致 + 时长甜点 + 持续在场 + 对称路径——五件事不需要新数据、新页面，只需要 dev 把"过渡"当作和"页面内容"同样重要的一等公民对待。

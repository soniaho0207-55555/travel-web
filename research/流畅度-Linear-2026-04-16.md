# research: 流畅度 - Linear - 2026-04-16

## 背景

Linear 是 2019 年成立的项目协作 SaaS（issue tracking / project mgmt），由 Karri Saarinen + Tuomas Artman + Jori Lallo 创立。从一开始就被作为"快得不正常的 issue tracker"卖。在工程师圈被当作 web app 流畅度的金标。

定位关键词：`familiar and fluid`、`opinionated`、`local-first`、`no spinners`、`structure felt not seen`。和 Jira / Asana / Notion 几乎是反方向——后者是 flexible，Linear 是 opinionated；后者重 chrome，Linear 把 chrome 弱化到几乎透明。

## 值得学的点

### 1. 流畅度的源头是架构，不是动效

Linear 流畅度不是靠"加更多动画"做出来的，是靠 sync engine 把"等服务器"这件事从用户视野里删掉。

机制：本地 IndexedDB 当真实数据库，所有 mutation 先在本地发生，UI 立刻反映；后台用 GraphQL + WebSocket 异步同步。结果是用户看到的每个动作都是同步的（实际是异步），看不到 spinner、看不到"提交中…"、看不到 disabled 按钮。

**核心 takeaway**：流畅度的最高表现不是"切换很丝滑"，而是**「该等的事情用户感觉不到」**。能在 16ms 内确认的事情就在 16ms 内确认；不能的，用乐观渲染骗过去。

### 2. "structure should be felt not seen"

设计 refresh 文章里反复出现这一句话——边框软化、圆角增加、彩色背景被取消。意思是：**结构要存在，但不能"举手"**。

具体做法：
- sidebar 整体 dim（降低饱和与对比）→ 让 content 区天然浮起来
- icon 数量减少、尺寸缩小
- tabs 从 full-width 改成 compact
- 用色彩"软"分隔代替线条硬分隔

**反义词**：每个 UI 元素都"举手喊看我"——按钮个个都是亮 primary、icon 全是彩色、tabs 全屏宽——这就是 Jira 1990s 路数。

### 3. "not every element should carry equal visual weight"

Linear 的视觉权重是**金字塔**：当前焦点最重，相邻状态半重，背景结构基本透明。这给了眼睛一个明确的"看这里"——你不用在 200 个等权元素里挑一个。

travel-web 的等价问题：列表里 20 个城市卡片是不是亮度、阴影、icon 用色全一样？如果是，眼睛会"扫不动"。

### 4. 8 条 Method 原则里和品味直接相关的几条

- **Build for the creators**：内部团队就是目标用户，痛得最深 → "我们自己都嫌慢"
- **Simple first, then powerful**：默认就是 power 用户路径，不靠"高级模式"开关
- **Decide and move on**：「make a decision, and move on」——产品决定不做投票
- **Don't invent terms**：「Projects should be called projects」——不发明 jargon

最后一条对 travel-web 也成立：城市页就叫"东京"，不叫"东京·全境探索之旅"。任何"创作者品味式命名"都是一种廉价。

### 5. Inter Display + LCH 色彩空间

字体：标题用 Inter Display（同字族但有更多表达），正文用 Inter Regular——不发明字体，用现成最好的。
颜色：从 HSL 迁到 LCH（人眼感知均匀的色彩空间）→ 主题之间过渡更"对眼睛诚实"，不会某一档突然"暗一截"。

这两条都在说：**品味不在炫技，在选用工业里最对的那个工具**。

## 不值得学 / 反而要避开的

### 1. Linear 的"无 spinner"是**昂贵**的

Sync engine 是工程团队 2-3 年从零搭的。travel-web 是静态站，不可能也不需要这套架构。

要偷的是**结果**："不让用户看到等"——具体落到 travel-web 上是：
- tap 立即给视觉反馈（press 态），不等导航完成
- 图片用渐进加载 / blur-up，不让用户看到"图正在出"
- 切页有过渡，不要白屏闪烁

而**不是**真的去搭一套 sync engine。

### 2. Linear 的 dim-everything 在 B 端 SaaS 里成立，在 C 端旅游里要小心

Linear 用户是回头率 100% 的工作工具用户，dim 的 chrome 不影响"我知道我在哪"。旅游 H5 是低频访客，第一次来——chrome 太弱可能让人迷路。

**适配判断**：travel-web 不是要"让 chrome 消失"，是要"让 chrome 服务于内容而不是和内容争"。

### 3. "Opinionated"在工具里是优势，在内容里是 risk

Linear 强 opinion 是"workflow 替你定好"。travel-web 如果对"该看什么"太 opinion，会变成强行投喂——和 Cereal 的"我们帮你删 7/8"是不同的事（Cereal 删的是噪音，travel-web 不能删的是用户兴趣）。

## 可迁移到 travel-web 的具体灵感

### 当前 travel-web 大概率有的 5 个流畅度问题

1. **Tap 无即时反馈**：H5 上 tap 一个城市卡片，多数实现是"等导航 → 新页加载完 → 才有视觉变化"。Linear 法是 1 帧内有 press 态（颜色变深 / 阴影下沉 / 微缩 0.97），让用户知道"按到了"。
2. **白屏跳转**：列表 → 详情多半是直接路由替换 = 白屏闪烁。Linear 法是 transition 或者至少保留旧背景 + 内容 fade-in。
3. **Loading 用 spinner**：图加载、详情加载用旋转 icon = 廉价。skeleton 是合格线，乐观渲染（占位骨架同步出现，图慢慢出）是 Linear 级。
4. **回退跳顶**：往下滚 5 屏 → 点进详情 → 回退 → 滚回顶。这是浏览器默认行为，但所有像样的 H5 都会保 scroll position。不保 = "你刚才看的不重要"信号。
5. **视觉权重一锅粥**：列表所有卡片亮度 / 阴影 / icon 用色一样 → 眼睛扫不出节奏。Linear 法是金字塔结构（推荐项更重 / 同质项 dim 些）。

### 不需要新功能也能做的 3 件

- 加 :active 态（CSS 一行）
- 加 view-transition / fade（现代浏览器 CSS）
- 替 spinner 为 skeleton（HTML + CSS）

**这三件不需要数据、不需要后端、不需要新页面，只需要 dev 一两天——但能在 3 秒内把"模板感"转成"做过手脚的产品感"**。

---

## 一句话总结

**Linear 的流畅度是"把'等'这件事从用户视野里删掉"。** travel-web 现在最大的问题大概率不在"动效不够"，而在"该有反馈的时候没有反馈"——tap 无 press 态、跳转白屏、loading 是 spinner、回退跳顶。这四件每件都让人 0.5 秒内感觉"这是个网页"，而不是"这是个产品"。

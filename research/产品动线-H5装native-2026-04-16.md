# research: 产品动线 - H5 装 native（拇指 / 微反馈 / 预测加载） - 2026-04-16

## 背景

上一次「产品动线 - app 流畅度」研究覆盖**多步骤路径**（共享元素、方向一致、过渡时长）。这次再下一层——**手指人体工学 + 微反馈分级 + 预测性加载**：决定 travel-web 这种 H5 是不是"装得像 app"还是"明显是网页"的关键三件套。

聚焦移动端，因为 travel-web 的核心场景是 375×812 的手机屏。所有 H5 项目最容易被识破"是网页"的瞬间，往往不是动效不够花，而是这三件没做：
- 主按钮放在拇指够不到的地方
- 按钮按下没有反馈分级（轻重不分）
- 滑到底才加载（必然出现 spinner）

## 三大支柱

### 支柱 1：拇指人体工学（Thumb Zone）

**Steven Hoober 1300+ 用户观察研究的核心数据**：
- 49% 用户**单手**持机
- 75% 屏幕交互靠**拇指**
- 拇指可达三区：
  - 🟢 绿区（easy reach）：屏底中央——拇指自然落点，0 拉伸
  - 🟡 黄区（stretch）：屏中两侧——可达但需伸展
  - 🔴 红区（hard reach）：屏顶角——需换手或双手

**对设计的强约束**：
- 主操作 / 主 CTA / 关闭按钮必须在 🟢 绿区
- 次要操作可以在 🟡 黄区
- 极少用 + 不重要 = 才能放 🔴 红区

**经典反例（Smashing Magazine 点名 Etsy 结账）**：
"The 'x' is at the edge of the thumb zone, forcing the user to stretch."
关闭按钮放在屏顶右角 = 单手用户每次必换手 / 必伸展 / 必心烦。

**iOS 原生骨架的反向证据**：iPhone 系统 app 90% 的主操作都在底部 tab bar——不是因为美观，是因为 Hoober 的研究就是这么设计指南的。Material Design 后来也跟进做 bottom navigation。

**Web/H5 落地**：
- Tab bar / 主 CTA 用 `position: fixed; bottom: 0`
- 关闭 X 按钮如果一定要放屏顶，就**同时**支持下滑关闭手势（modal 标准）
- 单手大屏（iPhone 14 Pro Max 6.7"）尤其要警惕：屏顶角对拇指几乎不可达

### 支柱 2：Touch Target 工业线 + 微反馈分级

**Touch target 最小尺寸（行业铁律）**：
- iOS HIG：**44×44 pt**（约 44pt = 44CSS pixel）
- Material Design：**48×48 dp**
- WCAG 2.5.5（无障碍）：**至少 44×44 CSS px**
- 跨平台保险：**48×48 px**

**相邻间距**：≥8px。挤在一起的小按钮 = 误点率激增。

**为什么这条不能妥协**：用户连续误点 1 次 = "这页面不靠谱"心智锁定，后续每次点击都带焦虑。研究表明 60+ 老年用户在 <44px 目标上错误率提升 30-50%——但即使年轻人，也会有"点不准"的微挫败感累积。

**微反馈分级（Apple HIG）**：

iOS haptic 三大类：
- `UIImpactFeedbackGenerator`（操作完成的物理感）：light / medium / heavy 三档
- `UINotificationFeedbackGenerator`（结果通知）：success / warning / error
- `UISelectionFeedbackGenerator`（选项切换的微滴）

**核心原则**（Apple 三条）：
1. **Harmony**：「it should feel the way it looks, and the way it sounds」——视觉、听觉、触觉一致
2. **Utility**：「Use haptic feedback only when you can provide clear value」——不滥用
3. **Hierarchy**：「the more important an action, the heavier the haptic」——分级

**反例**：每个按钮按下都震一下 = 廉价感（"震个没完"），用户烦到关掉系统震动。

**H5 没有 haptic API（除少数 navigator.vibrate）= 必须用视觉等价物**：

| 操作类型 | iOS haptic | H5 视觉等价 |
|---------|-----------|------------|
| 选项切换、tab 切换 | selection | 颜色 / 下划线 80ms 内变化 |
| 按钮按下确认 | impact light | `:active` 颜色加深 + 缩放 0.98（150ms） |
| 重要操作（提交 / 收藏） | impact medium | 缩放 + 阴影下沉 + 颜色变化（200ms） |
| 破坏性操作（删除 / 退出） | impact heavy | 短闪烁 / shake / 全屏微反馈（300ms） |
| 成功完成 | notification success | 绿色对勾 + scale up 短动画 |
| 出错 | notification error | 红色 + shake（200ms 内 3 次） |

**关键洞察**：travel-web 大概率所有按钮反馈一致（或干脆没反馈）= 信息扁平。即使没法做 haptic，也要靠**视觉的轻重**让用户感知"这个操作很轻"vs"这个操作有分量"。

### 支柱 3：预测性加载（Predictive Prefetching）

TikTok 的"无限丝滑"是怎么做的（Mux 拆解 + 行业共识）：

**核心机制**：
1. 用户当前在看视频 N，**视频 N+1 已下载完毕**，N+2 在下载，N+3 已发起请求
2. 滑动手势开始的瞬间，N+1 已经准备好播放——感知上"即时"
3. 根据 scroll velocity 动态调整 prefetch 距离：快速划者预取更多

**核心原则**：**用户能感知到的"等"，必须发生在"自然停顿点"**——
- ✅ 自然停顿：首次进入 app（用户预期等）/ 主动 pull-to-refresh / 提交表单
- ❌ 不能等：连续滑动中 / 浏览流里 / 详情页二次点击中

**这条原则把所有 loading 状态分成两类**：
- **可见 loading**：spinner / skeleton / 进度条 → 只在自然停顿点出现
- **不可见 loading**：prefetch / preload / preconnect → 用户看不到时悄悄发生

**Web/H5 实现工具**：
- `IntersectionObserver`：监听元素进入视口前 N px → 触发 prefetch
- `<link rel="prefetch" href="...">`：浏览器空闲时下载下一个页面 / 资源
- `<link rel="preload" as="image" href="...">`：当前页关键资源
- `loading="lazy"` / `loading="eager"`：图片懒/急加载分级
- Service Worker + Cache API：高级方案

**travel-web 大概率的痛点**：
- 城市列表向下滑动到底部才开始加载下一批 → 必然 spinner
- 详情页点开后才开始 fetch 数据 → 详情页空白 1-2 秒
- 图片是 native lazy loading 但没设 `aspect-ratio` → 加载时 layout shift

**改起来不难**：
- 用 IntersectionObserver 监听"还有 3 张就到底"，提前 prefetch 下批
- 列表卡片 hover / focus 时 `<link rel="prefetch">` 详情页 HTML
- 图片占位用 `aspect-ratio` 提前撑开高度，避免抖动

## 不值得学 / 反而要避开的

### 1. "把网页 wrap 成 webview 就是 app"

很多 H5 项目最后选择套个原生壳，希望用 native bridge 解决流畅度。但用户的眼睛比想象中锐利——只要 scroll 不是惯性回弹、tap 没有 haptic、过渡不是 push 动画——再装也是 H5。**与其装 native，不如把 web 能做的部分做到极致**。

### 2. 滥用 haptic（哪怕能用）

iOS Safari 16+ 部分支持 vibrate，Chrome Android 支持 navigator.vibrate。但**滥用比不用更糟**——每次按钮都震 = 廉价感。Apple HIG 明确说：「avoid overuse on minor or frequent actions」。

travel-web 上**几乎不该有 haptic**：内容站不是输入密集型工具，按钮不是高频操作，没有"破坏性"动作。如果一定要加，只在收藏 / 完成搜索 / 错误提示三个点用 light——别的地方一律视觉反馈足够。

### 3. 预测加载到所有页面

TikTok 能 prefetch 是因为 scroll 是线性、内容是预测可知的（下一个 video 就是 N+1）。travel-web 的探索动线是**非线性**——用户可能从 city A 跳到 city Z，不一定按列表顺序。

**只 prefetch "高概率下一步"**：
- 列表里**视口内**的城市详情 HTML（5-7 个最多）
- **当前**详情页的下一组图（不是所有图）
- **不**预加载整站（带宽和电量浪费）

## 可迁移到 travel-web 的具体灵感

### 当前 travel-web 大概率有的 5 个"明显是 H5"的破绽

1. **关闭按钮在屏顶右角**：modal / overlay 的 X 按钮放在 top-right = 单手用户每次必伸手
   → 改：放底部，或同时支持下滑关闭

2. **按钮 padding 不够**：CSS 看起来 16px 已经"够大"，但 touch 区域算上 padding 才 36-40px → 没到 44pt 工业线
   → 改：min-height: 44px，min-width: 44px 强约束

3. **所有按钮反馈一致**：tap → 颜色加深 → 完。没有"轻重"分别
   → 改：主 CTA 加 scale(0.98) + 阴影；普通按钮只颜色变；破坏性按钮加 shake

4. **滑到底才加载下批**：用户看到 spinner 那 1 秒 = 节奏断
   → 改：IntersectionObserver 监听倒数第 3 张卡片，提前 prefetch

5. **图片加载抖动**：图加载完才撑开高度，前面的卡片往下"跳"
   → 改：所有图加 `aspect-ratio: 16/9`（或对应比例），位置先撑开

### 三件 dev 一两天能改完的事

- 主操作位置审计 + 拇指可达性 review（CSS 不动布局，先标红 5 个高频按钮）
- 全站统一 touch target 最小尺寸 + 反馈分级 token（CSS variable）
- 列表 + 详情 prefetch 接入（IntersectionObserver + `<link rel="prefetch">`）

---

## 一句话总结

**H5 不该装 native，但该把"装不像"的破绽全消掉。** 拇指绿区 + 44pt 工业线 + tap 反馈分级 + 预测加载——这四件做了，用户在 5 秒内不会想到"这是网页"；不做，用户在 5 秒内一定意识到"这不是 app"。和 Cereal、Linear、动线流畅一样：流畅度的最高表现不是"加东西"，是**消掉所有让人分心的破绽**。

# research: 图片 - 渐进加载美学对比 - 2026-04-19

## 背景

Plan B P0 第 4 条，也是图片这一簇研究的收官。B.1-B.3 已覆盖"迁 CDN / ORB 归因 / 格式策略"。最后这条研究问：**图片在加载那 500ms~3s 里，用户看到什么？**

这是纯 taste 层的问题。加载完最终图固然是 1，但**加载过程那段空档**是 travel-web 和小红书/OTA 划界的关键——空档里出什么，决定了用户的"2% 不对劲"感。

四种主流策略：
1. **Skeleton loader**（灰色占位条，Facebook 风）
2. **LQIP / blur-up**（低质量缩略图 + 模糊过渡，Medium 风）
3. **Dominant Color**（纯主色占位，Google Images / Pinterest 风）
4. **BlurHash / ThumbHash**（编码成 20-30 字符的 hash 值再解码）

本研究逐个拆，再做 taste 排序，给 travel-web 的推荐。

## 1. 四种策略一览表

| 策略 | 代表产品 | 占位文件大小 | 是否需 JS | 视觉效果 | Taste 档次 |
|---|---|---|---|---|---|
| **Skeleton Loader** | Facebook / LinkedIn | 0（纯 CSS）| 否 | 灰白方块 + shimmer 动画 | **中**（实用但无个性）|
| **Dominant Color** | Google Images / Pinterest | ~50 bytes（CSS 变量）| 否 | 纯一色填充 | **中上**（克制但缺结构信号）|
| **LQIP / blur-up** | **Medium / Unsplash** | ~200 bytes（Base64 SVG）| 可选 | 模糊低清图 | **上**（能看出内容轮廓）|
| **BlurHash** | Wolt、部分产品 | 20-30 字符 | **是** | 模糊色块，JS 解码 | **中**（依赖 JS 是死穴）|
| **ThumbHash** | 更新一代（带 alpha）| 20-30 字符 | **是** | 同 BlurHash + 透明支持 | **中**（同上）|
| **CSS-only LQIP** | 2023+ 新玩法 | **0 额外请求**（写在 style 里）| 否 | 模糊渐变 | **上上**（最新 taste）|

## 2. Skeleton Loader · 最基础，也最容易翻车

### 2.1 是什么

灰色方块 + 可选 shimmer 微动效，模仿将要出现的内容结构。

### 2.2 优点

- 0 请求（纯 CSS）
- 告诉用户"一个卡片即将出现"
- Facebook 2015 年引入后成为行业常识

### 2.3 问题

- 灰色方块和旅行网站的"杂志编辑感"气质冲突——**看着像后台 admin 页面**
- 每个卡片都一样灰，用户看不出"它是一张巴黎图 vs 京都图"
- 加载完图出现时的**跳变**很突兀（灰到彩色，视觉冲击）

### 2.4 何时用

UX 研究显示 **< 0.5s 的加载不要用 skeleton**（太快闪现反而破坏感）。建议只在 >3s 的慢加载场景用。

### Taste 判断

> 对内容型网站（travel-web、杂志站、博物馆站），skeleton **是备选不是主选**。如果 hero 图加载 >3s，说明 CDN 没做好，修 CDN 比加 skeleton 更根本。

## 3. LQIP / Blur-Up · Medium 的经典范式

### 3.1 Medium 的做法（2015 年起）

用 José Pérez 博客的拆解：

- **缩略图**：从原图生成 **27×17 px 的小图**
- **质量**：JPEG quality 20%
- **文件大小**：约 **2 KB**
- **过渡**：在 canvas 上应用高斯模糊 → 等原图加载完 → 平滑替换

### 3.2 Mux 的对比数据

- JPEG LQIP：~200 bytes（16×16 px）
- WebP LQIP：~100 bytes（16×16 px）
- 作为 `<img>` 的 placeholder，base64 编码后嵌入 HTML

### 3.3 实现方式

**方式 A · inline data URL**（无额外请求）：
```html
<img
  src="data:image/jpeg;base64,/9j/4AAQSkZJRg..."  <!-- 200 bytes LQIP -->
  data-src="hero-full.jpg"
  style="filter: blur(20px);" />
<!-- JS 监听 data-src 加载完后换 src 并去掉 blur -->
```

**方式 B · SVG with Gaussian blur filter**（纯声明式）：
```html
<svg viewBox="0 0 1 1" preserveAspectRatio="none">
  <filter id="b"><feGaussianBlur stdDeviation="20" /></filter>
  <image href="data:image/jpeg;base64,..." filter="url(#b)" />
</svg>
```

### 3.4 视觉效果

用户看到：
1. 模糊的色块，能看出"这是一张风景""这是一张街景"
2. 0.5-2s 后清晰图平滑过渡出现

相比 skeleton 的灰方块，**多出一层"内容预告"** ——用户不是等"某张图"，是等"这张具体的图"。

### Taste 判断

> Medium 级体验的核心不是"有占位"，是"**占位有内容信号**"。用户看到模糊色块就能猜"巴黎夜景 / 京都街角"——这种预感让等待变成期待，不是焦虑。

## 4. Dominant Color · Google Images 的简约派

### 4.1 是什么

提取原图的**主导色**，加载前用这个色做纯色填充。

### 4.2 Google Images / Pinterest 的做法

- 服务器提取图片主导色（ColorThief 算法 / 火山 ImageX 自动给）
- CSS 变量或 inline style：`background-color: #a3b2c4`
- 加载完后纯色淡出，真图淡入

### 4.3 优缺

优点：
- 极简（几 bytes 的色值，无任何图片数据）
- 视觉上"安静"——没有模糊感干扰注意力
- 移动端加载尤其合适（数据量几乎为 0）

缺点：
- 无结构信号（用户不知道是风景还是街景）
- 多图场景下，所有卡片都是一种色的单色块，视觉单调

### Taste 判断

> Dominant Color 适合 **Google Images / Pinterest 这种"网格大规模图片展示"场景**。对 travel-web 的城市 hero 图（单张大图），相比 LQIP blur-up 有点弱——缺少"内容预告"的 taste 点。

---

## 5. BlurHash / ThumbHash · 看似高级，实则糟糕

### 5.1 是什么

把图片编码成 **20-30 字符的 hash 字符串**，前端用 JS 解码成模糊图。

### 5.2 致命问题（Mux 作者的犀利评论）

> "If you transform the image to a blurhash and back to an image, then send the image to the customer, what you've done is treat BlurHash as a preprocessing filter that loses the majority of its benefits."

**翻译**：如果你把 hash 解码成图再发给用户，那 BlurHash 压缩成 20 字符的优势等于 0（因为发给用户的是解码后的大图）。**唯一合理的用法是让客户端 JS 解码**——但这意味着必须等 JS 加载完才能显示占位。

而用户是来看图的，**等 JS 解码再显示模糊占位，本末倒置**。Base64 转换后体积膨胀 10 倍，压缩优势彻底消失。

### Taste 判断

> BlurHash / ThumbHash 是"技术上很酷"但**用户体验上错位**的方案。除非整个 app 是重 JS 架构（SPA + 大量前端状态），否则不推荐。travel-web 是内容型站，**不适合**。

## 6. CSS-only LQIP · 2023+ 的 taste 顶级

### 6.1 Lean Rada 的天才实现

单个 CSS 变量 `--lqip: 192900`（20 bit 整数），CSS 用 `calc()` + 位运算解码成 6 个径向渐变。

```html
<img src="hero.jpg" style="--lqip:192900">
```

对应的 CSS（一次写好，全站复用）：

```css
[style*="--lqip:"] {
  /* 用 calc() 和 mod() 提取 20 bit 里的 9 个分量 */
  /* 渲染 2×3 网格的径向渐变 */
  background: /* 一串 radial-gradient() 组合 */;
}
```

### 6.2 优点

- **0 请求、0 JavaScript、0 base64 污染 HTML**
- 渲染无延迟（CSS 原生）
- 单个字符串就是占位数据，写在 style 里

### 6.3 限制

- 模糊程度极高（6 个色块分布）
- 需要服务器端能生成这个 20 bit 编码
- CSS 写法复杂（但只写一次）

### Taste 判断

> CSS-only LQIP 是目前最**克制而优雅**的占位方案。几乎是 "占位技术的 endgame"。travel-web 如果能落地，品牌感能与 Vercel / Linear 对标。

## 7. 四种策略的 taste 排序（research-ux 口味）

```
档次从高到低：

1. CSS-only LQIP（2023+ 新玩法）——极简 + 0 请求
2. LQIP blur-up（Medium 经典）——内容预告信号强
3. Dominant Color（Pinterest 风）——安静，适合网格
4. Skeleton Loader（Facebook 风）——实用，但和编辑感冲突
5. BlurHash / ThumbHash——看似高级，实际体验差

以下彻底不推荐：
- 菊花 spinner（永远是廉价信号，见 🟡 段已有镜头）
- 白屏等待（任何时候都是最差）
```

## 8. 对 travel-web 的具体建议

### 8.1 MVP 阶段（本周能做）

用 **dominant color**：
- 在 js/data.js 里给每张图加 `dominantColor: "#a3b2c4"`（可以用 node-vibrant 离线批量跑）
- CSS 设 `.hero-placeholder { background-color: var(--dominant-color); }`
- `<img>` 加载完后淡出 placeholder

预期效果：所有图加载空档有主色打底，不是白屏；成本几乎为 0。

### 8.2 V2 阶段（2-4 周）

升级到 **LQIP blur-up**：
- CDN（腾讯 COS / 火山 ImageX）批量生成 16×16 WebP
- HTML 里 `<img src>` 用 base64 LQIP，`data-full` 放真实 URL
- JS 监听加载完替换，带 CSS transition 去模糊

预期效果：达到 Medium / Unsplash 的 taste 水准。

### 8.3 V3 阶段（有工程能力时）

迁移到 **CSS-only LQIP**：
- 服务端生成 20 bit 编码（Lean Rada 的代码是 MIT 开源）
- 写进 `data.js` 或 HTML style
- 全站去 JS placeholder 逻辑

预期效果：品牌感与 Vercel 级平台对标。

## 10. 局限与警示

- **CSS-Tricks 经典文章没抓到**：那篇是 blur-up 的西方正典，但页面返回空。本研究靠 José Pérez 的原始博客 + Mux 综合评测补。
- **Unsplash 具体拆解没抓到**：plan.com 博客 socket 断。Unsplash 的体验众所周知是 LQIP blur-up 但具体参数细节缺。
- **BlurHash 在特定场景仍有价值**：如果产品是重 SPA 架构 + 后端发 JSON 不发 HTML（比如原生 app 的 API 响应）—— BlurHash 的 hash 字符串体积小、可以放进 JSON。但 travel-web 是 SSG/SSR 架构，不适用。
- **移动网速越来越快 ≠ 可以省略占位**：5G 普及但国内真实网速有巨大地区差异。占位策略永远应该按"慢网络下的体验"设计，不是"理想网速下"。
- **CSS-only LQIP 的浏览器支持**：依赖 `calc()` + `mod()` + 径向渐变的复杂运算。微信 X5 内核的兼容性**未实测**，travel-web 真用时要测一次。

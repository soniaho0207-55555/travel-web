# research: 图片 - 国内 CDN 对比 - 2026-04-18

## 背景

travel-web 最痛的一个已知问题：**图片大陆可用性**。多轮 UX 反馈提到 Hero 图黑屏 / 景点图 ORB 拦截（京都、吴哥窟等），memory 里 `project_cdn_migration.md` 把"图片迁国内 CDN"标为 P0——排在微信小程序开发之前。但 research-ux agent 过去 8 轮研究**完全没有**碰这个维度。

本研究是 Plan B P0 的第一条：**把国内主流 CDN / 对象存储在 H5 图片场景的 taste + 实战表现做横评**，为 travel-web 后续的 CDN 迁移提供决策依据。

**纪律提醒**：本研究是 research-ux 视角——只关心"品牌感 / URL 美学 / 图片处理灵活度 / 格式策略"这些 taste 层和前端体验层的东西。**价格只是次要信号，不做深入比价**（那是 PMO / CEO 的决策维度）。

## 2. URL 品牌感横评（这是 research-ux 最该看的维度）

URL 是用户看不到但**开发者每天看 + 泄露时用户看得到**的东西。URL 长什么样，暗示品牌气质。

### 2.1 默认域名对比

```
七牛云 默认：  http://xxx.bkt.clouddn.com/image.jpg
又拍云 默认：  http://xxx.b0.upaiyun.com/image.jpg
阿里 OSS：    https://xxx.oss-cn-hangzhou.aliyuncs.com/image.jpg
腾讯 COS：    https://xxx.cos.ap-shanghai.myqcloud.com/image.jpg
火山 ImageX： https://xxx.volccdn.com/image.jpg~tplv-abc-resize:800.webp
```

**Taste 判断**：
- 七牛 / 又拍的默认域名**最丑**（qiniucdn / upaiyun / bkt / b0 都是 backend 术语暴露）
- 阿里 / 腾讯的默认域名**企业感**（oss-cn-hangzhou、cos.ap-shanghai、myqcloud——地区 + 产品线全暴露）
- 火山的域名**相对清爽**（volccdn），但 URL 后缀 `~tplv-xxx` 是行业丑写法

**共同结论**：**所有国内 CDN 的默认域名都不能对外用**。必须绑定自定义域名（`img.travelweb.com` 或类似），否则品牌感立刻崩。

### 2.2 图片处理参数 URL 风格

这是前端写 `src` 时每天都看的东西，**最影响开发者审美**。

**阿里 OSS**（query string 风格）：
```
https://img.travelweb.com/kyoto.jpg?x-oss-process=image/resize,w_800,h_600,m_lfit/quality,q_80/format,webp
```
- 优点：参数结构化，像函数调用
- 缺点：长，`x-oss-process` 这个前缀很丑

**腾讯 COS / 七牛云**（path 风格）：
```
https://img.travelweb.com/kyoto.jpg?imageView2/1/w/800/h/600/q/80/format/webp
```
- 优点：比阿里短
- 缺点：`imageView2` 这个版本号很技术味

**火山 ImageX**（template 风格）：
```
https://img.travelweb.com/kyoto.jpg~tplv-abc123-resize:800:600.webp
```
- 优点：最短、最像"正常 URL"
- 缺点：`~tplv-abc123` 那串模板 ID 很难看

**Taste 判断**：
> **没有一家的 URL 是真的好看的**。阿里最结构化（适合系统），七牛腾讯折中（适合 CMS），火山最紧凑（适合高流量）。travel-web 如果追求"URL 是不是泄露时也能看"的极致 taste，需要在 CDN 之外加一层**自己的反代/image server**（Next.js Image / Cloudflare Workers 之类）把 URL 重写成 `img.travelweb.com/kyoto/hero?w=800` 这种干净格式。

## 3. 图片处理 API 的灵活度

直接决定前端能不能"一条 URL 动态做所有事"——是否需要后端预生成多尺寸。

### 3.1 阿里 OSS（参数最丰富）

```
/resize,w_200                  // 宽 200
/resize,h_200                  // 高 200
/resize,p_50                   // 百分比 50%
/resize,m_fill,w_200,h_200     // 填充居中裁剪
/resize,m_pad,w_100,h_100,color_FF0000  // 填充颜色
/quality,q_80                  // 相对质量
/quality,Q_80                  // 绝对质量
/format,webp                   // 格式转换
```
5 种 scaling 模式（lfit / mfit / fill / pad / fixed），像素限制 16384，最细粒度。

### 3.2 腾讯 CI（模式化，6 种）

imageView2 支持 0-5 六种模式：
- 0 等比缩放不裁剪
- 1 最小尺寸 + 居中裁剪（常用）
- 2 最大尺寸
- 3 最小尺寸不裁剪
- 4 / 5 长短边约束

限制：静态图 ≤32MB，尺寸 ≤50000 px，总像素 2.5 亿。

### 3.3 火山 ImageX（模板化）

**和前两家不同**：火山的图片处理是**预先定义模板**（tpl），URL 里引用模板 ID——不是每次在 URL 里写参数。

优点：URL 短、可缓存友好、防篡改（用户不能随便改参数）
缺点：新尺寸要去后台建模板，灵活度低

### 3.4 七牛 Kodo（imageView2 + imageMogr2）

和腾讯 CI 高度相似（imageView2 语法几乎一致，因为腾讯挖了一批七牛的工程师）。除了 imageView2 外，还有更强大的 imageMogr2（支持旋转、模糊、锐化等），但语法更复杂。

### Taste 判断

> **前端爱谁恨谁**：
> - 爱**阿里 OSS**：参数最全、最像"标准 API"、文档最细
> - 爱**腾讯 CI / 七牛**：URL 短、参数简洁、一行搞定常见场景
> - 恨**火山 ImageX**：每个新尺寸都要去后台建模板，灵活度太低——但如果 travel-web 的图片尺寸**是固定几组**（hero / card / thumbnail / gallery），模板化反而是优点

## 4. 图片格式策略 — WebP / AVIF / HEIC

### 4.1 压缩率对比（来自火山博客实测）

```
HEIC < AVIF < WebP < JPEG < PNG
└── HEIC 体积最小（约 JPEG 的 1/8）
```

AVIF 实测：45KB JPEG → 30KB AVIF（**节省 33%**），下载速度从 256KB/s 提升至 10MB/s（博客园评测）。

### 4.2 浏览器兼容性（国内场景）

| 格式 | iOS Safari | Android Chrome | 微信 WebView | 老旧设备 |
|---|---|---|---|---|
| **WebP** | ✅ 14+ | ✅ 全系 | ✅ | ✅（基本全兼容）|
| **AVIF** | ✅ 16+ | ✅ 新系 | ⚠️ 新版本 | ❌ |
| **HEIC** | ✅ 原生 | ⚠️ 需解码 | ⚠️ | ❌ |

### 4.3 Accept header 自适应（火山独有的亮点）

火山 veImageX 的一个真正 taste 点：**根据 HTTP Accept header 自动返回最优格式**。
- 浏览器 Accept 含 `image/avif` → 返回 AVIF
- 只含 `image/webp` → 返回 WebP
- 都不含 → 返回 JPEG

一个 URL，多种格式，自动降级。其他四家基本要前端用 `<picture>` + srcset 手写 fallback。

### 4.4 `<picture>` + srcset 写法（其他四家必须用）

```html
<picture>
  <source srcset="kyoto.jpg?x-oss-process=image/format,avif" type="image/avif">
  <source srcset="kyoto.jpg?x-oss-process=image/format,webp" type="image/webp">
  <img src="kyoto.jpg" alt="京都">
</picture>
```

### Taste 判断

> **格式策略的 taste 点**：
> - **傻瓜路线**（火山 ImageX）：不用前端改代码，CDN 按 Accept 自适应——**travel-web 的小团队最适合这条**
> - **灵活路线**（阿里 / 腾讯 / 七牛）：前端写 `<picture>` 手动兜底——控制精细但代码累
> - 任何情况下，**WebP 必须开启**（兼容性已经全面，节省 40-60% 流量）
> - AVIF 是"锦上添花"，老设备不支持时自动降 WebP 即可
> - HEIC 目前只在 iOS 原生相册有意义，**Web 场景不推荐**（Android/微信兼容性差）

## 5. 免费额度与个人/小团队视角

这部分有实战价值，但属于运营/成本决策，简要列一下：

| CDN | 个人站（月流量 <50GB） | 小团队站（月流量 50-500GB） | 成长站（月流量 500GB+） |
|---|---|---|---|
| 七牛云 | 免费（10GB 以内）| ¥0.24/GB | 谈判价 |
| 又拍云 | 免费（10GB + HTTPS 免费）| ¥0.25-0.3/GB | 联盟用户优惠 |
| 阿里 OSS | 基本免费 | ¥0.25-0.5/GB（闲时/忙时）| 企业折扣 |
| 腾讯 COS | **6 个月 50GB 免费** | ¥0.28/GB | Bilibili/知乎/小红书 同款 |
| 火山 ImageX | **10GB 流量 + 20TB 处理量/月免费** | 1元 100GB 流量促销 | 字节系生态 |

**小团队 / MVP 最优选**：火山 ImageX 或腾讯 COS。
**taste + 成熟度最优选**：阿里 OSS + 自己反代。

## 6. 对 travel-web 的具体建议

基于六个 research-ux 目标里的 **"图片美观且大陆可用"**，推荐路径：

### 6.1 最小可行方案（MVP，立刻能做）

- **选腾讯 COS 或火山 ImageX 之一**（都有足够免费额度覆盖 travel-web 体量）
- **开自定义域名**（`img.travelweb.com` + CNAME）—— 必须，别用默认域名
- **启用 HTTPS**（免费证书）
- **开 WebP 自动转换**
- **删掉所有 Google Photos / Unsplash 等墙外图床链接**（这是 ORB 黑屏的根因）

### 6.2 升级方案（有人力时）

- 迁到**火山 ImageX**——唯一原生支持 Accept-based 格式自适应的，travel-web 的前端代码可以最少
- 所有城市的 Hero 图按模板预生成 3 种尺寸（mobile 750 / tablet 1080 / desktop 1920）
- 启用 AVIF fallback WebP fallback JPEG 三级降级
- 加占位策略（见 B.4 研究）

### 6.3 终极方案（有工程能力时）

- 自建 image server（Cloudflare Workers / Next.js Image / 自己写 Go）
- 在 CDN 之上加一层 URL 重写，把 `xxx.cos.myqcloud.com/image.jpg?imageView2/...` 重写成 `img.travelweb.com/kyoto/hero?w=800`
- **URL 从"技术 API 字符串"升级为"内容资源地址"**——这是 Stripe / Apple / Linear 级 taste 的标志

## 7. 本次研究可直接产出的镜头候选（3 条）

**放入 🔵 段（美学/流畅度/惊喜）**：

1. **图片 URL 品牌感检验**：URL 里暴露 `oss-cn-hangzhou` / `myqcloud` / `bkt.clouddn` / `b0.upaiyun` 这类 backend 术语 = 没做自定义域名绑定；`img.brand.com/...` = 有基本品牌感；`img.brand.com/{slug}?w=800` 无处理后缀 = Stripe 级 URL 美学
2. **图片格式兜底**：`<picture>` + srcset 有没有 AVIF / WebP / JPEG 三级降级？只有 JPEG = 浪费 40-60% 流量 + 加载慢
3. **图片域名是否国内**：控制台 Network 面板看图片 host，是 `*.myqcloud.com` / `*.aliyuncs.com` / `*.volccdn.com` / `img.brand.com` 还是 `googleusercontent.com` / `unsplash.com`？后者 = 墙外源，大陆可用性赌运气

## 8. 局限与警示

- **本研究聚焦 taste + 前端体验**：对 CDN 的稳定性、企业级特性、数据安全、合规（备案）没展开——这些是 PMO / 运维 的决策维度。
- **价格数据时效性**：国内 CDN 价格每半年可能有调整，具体数字请以官网为准。
- **未覆盖**：七牛 developer 文档因 enterprise block 未能深入抓取，七牛的高级处理能力（imageMogr2、AI 图像处理等）在本研究里偏薄。
- **没有做实测**：本研究全部基于文档和社区评测。真正迁 CDN 前应该**选 1-2 家做 A/B 测试**（同一张图在不同 CDN 下的加载时长、首屏 LCP、不同地区延迟）。
- **"备案"是硬门槛**：所有国内 CDN 都要求**域名先在工信部备案**（travel-web 如果没备案，选哪家都用不了），这是一次性 20-30 天的流程。如果 CEO 还没处理，这条是最前置的 blocker。

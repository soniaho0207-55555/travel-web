# research: 美学 - Things-3 - 2026-04-17

## 背景

Things 3 是 Cultured Code（斯图加特）出品的 todo/任务管理器，两次获 Apple Design Award。2017 年从 Things 2 彻底重建，被广泛视为"极简设计与功能完整性平衡"的标杆。研究目标：拆解它的**信息层级克制**和**渐进披露**模式，为 travel-web 的景点卡片、时间轴、详情页提供 taste 参照。

---

## 1. 核心设计哲学：Restraint（克制）

Things 3 的设计不是"做减法"后的空洞，而是"每个可见元素都赚到了它的存在权"。

- **Design is not an afterthought — it's a way of building apps**（Cultured Code 官方定位）
- 评论者反复用一个词：**restraint**——不是功能少，而是不该看到的东西确实不在视野里
- 对比：Todoist 在同一屏幕密度下塞入标签/优先级/指派人/评论数，Things 只展示文字 + 一个圆形 checkbox

### Taste 提取

> 克制不是删功能，是删"默认可见"。功能依然在，但只在用户伸手时出现。

---

## 2. 渐进披露（Progressive Disclosure）——Things 3 的杀手级模式

这是 Things 3 最值得 travel-web 学习的单一模式：

### 2.1 任务打开 = 白纸变形

点击一条 todo，它不是跳到新页面，而是**原地展开（smooth transform）**——从一行文字变成一张"干净的白纸"。标签、checklist、开始日期、截止日期等 detail fields **"neatly tucked away in the corner until you need them"**。

- 展开前：仅标题 + checkbox（1 级信息）
- 展开后：标题 + notes + metadata 入口（2 级信息，但 metadata 仍折叠）
- 主动点击 metadata 区域：日期/标签/checklist 出现（3 级信息）

三层渐进，每层只在用户主动触发时才露面。

### 2.2 Magic Plus 按钮

浮动操作按钮（FAB），但不是固定功能——**拖动方向决定行为**：

- 点击 = 当前位置新建 todo
- 向上拖 = 新建 heading
- 拖到侧栏某个 project = 直接在那个 project 建 todo

一个按钮，三种行为，零额外 UI 元素。

### Taste 提取

> 信息层级不是用不同字号/颜色分出来的，而是用**时间**分出来的——先看到什么、后看到什么。

## 3. 留白策略

Things 3 被反复描述为 **whitespace-forward**：

- 任务列表区域：左右 margin 大于行业平均，每条 todo 之间有可感知的呼吸间距
- 空状态设计：列表为空时不是"暂无任务"灰字，而是大面积白纸——**空也是设计**
- 侧栏 icon 之间的间距与内容区一致——没有"侧栏挤、内容区松"的割裂

但有一个真实局限：**当任务量极大时（>50 条），留白让垂直滚动变长，被用户比喻为"电话簿感"**。这说明留白策略需要配合折叠/分组才能在密度场景下存活。

### Taste 提取

> 留白的代价是滚动距离。解药不是缩减留白，而是加分组折叠——让单组内的留白保持，但组与组之间可以收起。

## 4. 排版与色彩

### 排版

- 系统字体（SF Pro），不自定义——把"字体性格"让位给"内容本身的节奏"
- 标题和正文之间靠 **字重（weight）** 区分，不靠字号跳级
  - 项目名：semibold
  - 任务标题：regular
  - 注释/meta：regular + 灰色
- 结果：页面扫一眼就知道谁是项目、谁是任务、谁是注释——靠 weight + color 的两轴，不靠 size 的第三轴

### 色彩

- 内容区几乎全白+黑字+灰 meta——**彩色只出现在两个地方**：
  1. 侧栏导航 icon（每个 section 一个颜色：蓝=Today, 黄=Upcoming, 绿=Logbook...）
  2. 用户自定义的 tag 颜色
- 这意味着：**颜色是导航锚点，不是装饰**。内容区拒绝颜色竞争

### Taste 提取

> 颜色的角色是"路标"（告诉你在哪），不是"装饰"（让页面好看）。内容区越素，路标越有效。

## 5. 动效哲学

Cultured Code 构建了自己的动画工具链（custom animation toolkit），核心原则：

- **动效是解释，不是表演**：任务展开/折叠用 transform 动画——用户看到的是"这条东西变大了"而不是"跳到了新页面"
- **Unfolding animations keep your place**：展开时维持用户在列表中的位置，不跳到顶部
- 拖拽排序时有弹性阴影跟随——让用户感知"这条正被我抓着"
- 没有 gratuitous motion（无意义装饰动效）

### Taste 提取

> 动效的最高标准：用户感知到"空间连续性"而不是"炫酷动画"。如果用户注意到动效本身，说明动效太多了。

## 6. 对 travel-web 的 taste 启示

| Things 3 模式 | travel-web 对标场景 | 应该看到什么 |
|---|---|---|
| 渐进披露三层 | 景点卡片 → 详情页 | 卡片只放名字+一句话+图；详情展开后才见票价/时间/地图 |
| 颜色=路标 | 主题标签/大洲标签 | 标签颜色是导航功能，不是装饰；内容区保持中性 |
| 字重>字号 | 城市名 vs 景点名 vs 介绍文 | 用 weight 分层，不要每级跳 4px 字号 |
| 空也是设计 | 搜索无结果/空城市 | 不要塞"推荐"/"热门"填空，给白纸和一句有味道的话 |
| 展开保位置 | 列表→详情→返回 | 返回后必须回到原 scroll 位置 |

---

## 7. 局限与警示

- **密度陷阱**：Things 3 在任务 >50 条时显得冗长——travel-web 如果城市/景点数量增多，纯留白策略会让首页变成无尽滚动。需要像 Things 的"Area > Project > Heading"分组机制来管理密度
- **单色内容区的代价**：Things 内容区几乎没有视觉惊喜——对 todo 可以（用户来干活不来欣赏），但旅游内容需要情感密度，不能全素。**需要在 Things 的克制框架里注入图片叙事的温度**
- **无社交/共享维度**：Things 是纯个人工具，没有"别人也在看"的社交信号——travel-web 需要克制地加入社会证明，但不能掉进 OTA 的"X 万人想去"陷阱

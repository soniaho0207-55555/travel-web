# research: 排版 - 移动端长内容自持极限 - 2026-04-19

## 背景（这是 P00000 修复的弹药库）

CEO 2026-04-18 拍了 P00000：**景点介绍"是什么"段塌方**——字太多、字太大、字体奇怪、没有分级分段。UX 2026-04-18 上线实测，给出了精确现场数据：

- 拴日石单段 **297 字 / 520px / 64% viewport / 2 次半屏 swipe 才能读完**
- 字号 17px、line-height 28.9px（1.70x ✅）、字体 `Playfair Display + Noto Serif SC`
- 之前一直以为的"斜体"其实是 **serif on CJK mobile 的视觉错觉**，`font-style: normal`
- 金阁寺用了分段 + 折叠结构（做对了），但 60% 景点没用

**本研究的目的**：把"移动端中文长内容怎么排才不被读者弃掉"做成可引用的知识库，为 Dev-H5 修复 P00000 提供权威弹药 + 精确数字。

## 1. 结论前置：UX 实测 vs 业界标准的对照

| 参数 | UX 实测（拴日石现场）| 业界最佳实践 | 诊断 |
|---|---|---|---|
| **字号** | 17px | CJK mobile body：**15-17px**，16 是甜点 | ⚠️ 偏高 |
| **行距** | 28.9px（1.70×）| CJK 1.5-1.8em | ✅ OK |
| **字体** | Playfair Display + Noto Serif SC | **CJK mobile body 强烈推荐 sans**（PingFang / 思源黑体）| ❌ 病灶 |
| **字距** | letter-spacing 0.51px | CJK body **必须 0**（西文 +tracking 规则不适用）| ❌ 病灶 |
| **单段字数** | 297 字 | 移动端单段 ≤ **120-150 字**（≈1 viewport）| ❌ 严重 |
| **行长** | 屏宽下约 30-32 汉字 | CJK 最佳 **≤40 汉字/行** | ✅ OK |
| **段落分隔** | 无 | 必须有（`\n\n` 或子标题）| ❌ 严重 |

**结论**：拴日石那段的塌方不是某一个参数错，是**三个病灶叠加**（serif + 字距 + 单段超长）+ **数据层未分段**。UX 提的 ROI 排序（字体 > 字号 > 字距 > 行距）和业界建议完全吻合。

## 2. 行长（字符数/行）——为什么 CJK 特殊

### 2.1 行长标准（来自 Baymard + WCAG）

- **拉丁文**：50-75 字符/行为最佳，最多 80（WCAG 1.4.8）
- **CJK**：**≤40 汉字/行**（中日韩三语共同）

### 2.2 为什么 CJK 要更窄

搜索聚合学术研究（ScienceDirect CJK mobile reading studies）的共同结论：

- **每个汉字 = 一个完整语义单元**（英文单词才是语义单元）
- 一行 40 汉字 ≈ 一行 60-80 英文字符的**信息密度**
- 汉字方块字形"**形状相似度高**"，眼睛扫描 vs 英文形状更累

### 2.3 实际影响

行太长会导致：
- 眼睛从行尾到下一行开头需要"大跳跃"，易串行
- 用户感觉"冗长压迫"，**跳读/放弃**
- 阅读速度和理解力双降（Baymard 实测）

行太短则：
- 眼睛频繁回返，节奏被打乱
- 读者可能"跳读"

### Taste 提取（针对 travel-web）

> 375px mobile 屏宽下，17px CJK body 恰好 30-32 汉字/行，**在 40 汉字上限内**——**行长这条 travel-web 是对的**。所以行长不是 P00000 的主矛盾，**字体 + 单段字数**才是。

## 3. 字号 — 2024 年的共识窗口

### 3.1 Learn UI Design 2024 推荐

- **Mobile body**：16-20px，**17 是起点**
- **二级文本**：body -2px（13-14px）
- **标题**：28-40px

### 3.2 CJK 特殊要求

多篇学术研究（搜索聚合）：
- **CJK mobile body 甜点：15-17px**
- <14px 在中文会模糊（笔画多）
- >18px 在单屏显示信息量太低

### 3.3 travel-web 的具体建议

| 位置 | 当前 | 建议 | 理由 |
|---|---|---|---|
| H1 城市名 | 28px | 保留 | 标题档对 |
| 景点 H2 | 未测 | 22-24px | 和 body 拉开档次 |
| chip 子标题 | 18px | 保留 | 和 body 拉开 2-4px |
| **body 正文** | **17px** | **→ 15.5-16px** | CJK mobile 甜点 |
| meta / caption | 未测 | 13px | body -2-3px |

**降 1.5px 的意义**：同一段 297 字，从占 520px（64% viewport）降到 ≈470px（57%）——约省 50px / 1.5 行。小幅改善，但配合字体 + 分段才能真治本。

## 4. 字体 · sans vs serif（P00000 的头号病灶）

### 4.1 UX 实测的关键发现

```
font-family: "Playfair Display", "Noto Serif SC", Georgia, serif
```

中文 fallback 到 **思源宋体（Noto Serif SC）**——笔画粗细变化 + 衬线 = **在 17px CJK body 上显得"像斜体"**（几何学正确：起笔捺笔有斜向笔画）。

### 4.2 业界共识：CJK mobile body 必须 sans

从多源汇总（搜索聚合 + Learn UI + 数英）：

- **Medium / Substack**：body sans-serif（Charter / Gothic）
- **微信读书 / 得到 / 多看阅读**：默认 sans（思源黑体 / 兰亭黑）
- **Bear / Notion / Things 3**：body 全 sans
- **杂志印刷仍用 serif**：但印刷 300 DPI，手机屏密度差一个量级

### 4.3 为什么 serif 在 CJK mobile 翻车

- **屏幕渲染精度不足**：衬线的细节在 2-3x DPR 下仍模糊
- **笔画粗细对比**：让眼睛在"粗线-细线-衬线"之间跳跃，阅读速度慢 10-15%（CJK 学术研究）
- **错视**：斜向笔画让中文显得"歪着"（即 UX 几轮反馈的"斜体"误判）

### 4.4 travel-web 建议的字体栈

**正文 body**：
```css
font-family: -apple-system, BlinkMacSystemFont, 
             "PingFang SC",         /* iOS / macOS 苹方 */
             "Noto Sans SC",        /* Android / Linux 思源黑 */
             "Microsoft YaHei",     /* Windows 微软雅黑 fallback */
             system-ui, 
             sans-serif;
```

**标题 H1/H2（保留 serif 制造层级对比）**：
```css
font-family: "Playfair Display",    /* 西文标题保留 */
             "Noto Serif SC",       /* 中文衬线保留 */
             Georgia, serif;
```

这种"**标题 serif + 正文 sans**"是 Medium / Substack / 《三联生活周刊》数字版都在用的经典搭配——**通过字体对比制造层级**，而非全篇 serif。

### Taste 提取

> Serif 在 CJK mobile body 不只是"不够 taste"，是**读着像翻译过来的学术报告**（UX 原话）。它破坏的是"编辑写给你"的亲密感，变成"让你证明你够勤奋"的姿态。travel-web 是内容型站，**让用户轻松读完 400 字比让页面"显得严肃"重要得多**。

## 5. 字距 — CJK 必须 0

### 5.1 UX 现场测量

```
letter-spacing: 0.51px
```

这是 0.5px 左右的正向字距（把字母拉开）。

### 5.2 为什么 CJK body 字距必须 0

从数英"中文版式的细节规范"引文：

> "内文保持为 0" / "勿调整内文字距（会破坏阅读方向识别）"

**西文字距** +tracking 的场景：
- 全大写 + 小号（CAPS OR SMALL CAPS）—— 有帮助
- Display 级标题 —— 营造空间

**但 CJK 字距+value 的后果**：
- 汉字方块间距被**人为拉开**
- 看着像"**小儿习字帖**"或"**拼音课本**"
- 破坏读者的词组识别能力（中文没有空格，靠字距决定"哪几个字属于一组"）

### 5.3 例外：字距可以是负值吗

部分设计师会给 CJK body 设 `letter-spacing: -0.02em` 来**收紧**（让字看起来更密）——这是设计师级操作，travel-web 不建议，太容易翻车。

### Taste 提取

> CJK body 字距 = 0 是**不可协商的默认**。UX 实测的 0.51px 来自 CSS 继承或主题默认值，这种"我们以为对西文有效的 tracking 规则"**在中文是毒**。

## 6. 单段字数上限 · 最直接的 P00000 killer

这是整个研究最重要的一条。

### 6.1 研究与实测对齐

| 来源 | 推荐单段上限 |
|---|---|
| Baymard Institute | 没给具体汉字数，但强调 "users perceive content as intimidating" |
| 设计达人（中文）| "一般情况下都不会超过 40 个汉字 / 行"，段落需拆分 |
| 数英 | 段落间距 0.5-1em，未说段落长度 |
| **UX 实测（travel-web 反向）** | **297 字 = 弃读警报**；**金阁寺分段后 200 字 / 80 字为可读** |

### 6.2 实用阈值（本研究建议）

```
移动端 CJK body 单段字数：
  < 80 字：短促，节奏快（用于钩子、强调）
  80-120 字：甜点（约 4-5 行 × 25 字 = 1 viewport 内读完）
  120-150 字：上限（5-6 行 = 恰好 1 viewport）
  > 150 字：警报
  > 250 字：**弃读风险高**
  > 300 字：**物理确认弃读**（UX 实测 297 字 = 2 次 swipe）
```

### 6.3 为什么 150 字是门槛

**viewport 视觉饱和度**：
- 375px mobile 屏、15.5px 字号、1.7 lh、25 字/行
- 约 26.35px 行高 × 5-6 行 = 131-158px 高度
- **恰好等于一个"注意力 chunk"**（约 1/5 - 1/4 viewport）

超过 150 字 ≈ 用户滚动前读不完这段 = 需要记忆+滚动+回顾 = **认知负担跳级**。

### 6.4 travel-web 的两级修复方案

**方案 A · 数据层修复（PM sweep）**：
- PM 过一遍所有 landmark.whyVisit 字段
- 按金阁寺模板拆：`{是什么}`（2-3 短段）+ `{为什么独特}`（1 段）+ `{跨文明}`（1 段）
- 每段 ≤150 字
- 工作量：70+ 景点 × 约 10 分钟/景 ≈ 12 小时

**方案 B · 渲染层兜底（Dev-H5 一次性）**：
```js
function splitLongParagraph(text, maxChars = 150) {
  if (!text || text.length <= maxChars) return [text];
  // 优先按"\n\n"切分（数据层已分的尊重）
  if (text.includes('\n\n')) return text.split('\n\n');
  // 再按"。"+ 长度阈值切分
  const sentences = text.split(/(?<=。)/);
  const paragraphs = [];
  let current = '';
  for (const s of sentences) {
    if ((current + s).length > maxChars && current) {
      paragraphs.push(current);
      current = s;
    } else {
      current += s;
    }
  }
  if (current) paragraphs.push(current);
  return paragraphs;
}
```

**推荐方案 B**：
- 一次代码，所有景点自动生效
- 未来新景点写的任何长段都会自动兜底
- PM 不需要做一次性 sweep（避免新景点又污染）

### Taste 提取

> 单段字数是 **P00000 的核心参数**。字体改了但段还是 297 字，照样弃读；段拆到 150 字但 serif 没改，仍然累。**必须同时做**。方案 B 的渲染兜底是一劳永逸的解——travel-web 用户数据模型和渲染分离得不错，这条路径 Dev-H5 几小时能出。

## 7. 段落分隔的视觉表现

### 7.1 两种中文段落分隔方式（来自数英）

**印刷书籍传统**：
```css
text-indent: 2em;  /* 首行缩进 2 个汉字位 */
```

**网页现代**：
```css
margin-bottom: 0.5em;  /* 段间空行 */
```

### 7.2 travel-web 现状

UX 描述金阁寺"做对了"——用的是**段间空白**+ **inline chip 子标题**方式。这是现代网页正确做法，不需要改为 `text-indent`。

### 7.3 推荐落地

- **段间距**：`margin-bottom: 0.75em`（约等于一行高）
- **子标题前间距**：`margin-top: 1.5em`（拉开层级）
- **子标题**：左金线 + 小号 eyebrow（UX 观察到的 L-06 修复已经是这个方向）

## 8. 盘古之白 · 中英混排的必做项

### 8.1 是什么

中文与英文/数字之间**自动插入空格**：

```
❌ 我用iPhone13拍了2张照片
✅ 我用 iPhone 13 拍了 2 张照片
```

名字来自台湾设计师 **盘古之白**（名字戏称"让盘古开天辟地般分开中英文"）。

### 8.2 为什么重要

- 中英紧贴看着挤，视觉不舒服
- 数字和汉字也该有间距

### 8.3 travel-web 的实现选项

**选项 A · JS 自动处理**：
- [pangu.js](https://github.com/vinta/pangu.js) 开源库，一行代码全站生效
- 适合动态内容

**选项 B · 数据层写入时加空格**：
- PM 写文案时手动 / 编辑工具辅助
- 可控但维护成本高

**选项 C · CSS `font-feature-settings`**：
- 部分中文字体（如 PingFang）有合字特性
- 实际支持率低，不推荐

**推荐选项 A**：pangu.js 一次接入，所有中英混排自动处理。

### 8.4 travel-web 现场观察

UX 反馈的 `"今天 到今天北京地铁..."` 有一个独立的空格——这个空格的来源不清楚，**可能已经在用 pangu.js 但覆盖不全**，也可能是数据手写加的。需要 Dev 排查。

## 9. Line-height 和字距的交互

UX 实测 line-height 28.9px（1.70x），和业界推荐完全对齐。**这是 travel-web 做对的事**——不需要动。

但要注意：
- **如果字号改为 15.5px，line-height 应变为 26.35px（仍保持 1.70）**
- **不要保留 28.9px 绝对值**，否则行距与字号比例失调（变成 1.86x）

```css
/* 正确 */
font-size: 15.5px;
line-height: 1.7;   /* 相对单位，随字号变化 */

/* 错误 */
font-size: 15.5px;
line-height: 28.9px;  /* 固定像素，字号变了这里也得同步改 */
```

## 10. 综合：P00000 的完整修复配方（给 Dev-H5）

把 UX 实测 + 本研究的所有结论整合成一份可直接 copy 的 CSS：

```css
/* travel-web 景点介绍页 body 正文 */
.landmark-body,
.landmark-whyVisit,
.landmark-overview {
  font-family: -apple-system, BlinkMacSystemFont,
               "PingFang SC", "Noto Sans SC", "Microsoft YaHei",
               system-ui, sans-serif;
  font-size: 15.5px;          /* ↓ 从 17px */
  line-height: 1.7;            /* ✅ 保持 */
  letter-spacing: 0;           /* ↓ 从 0.51px */
  color: rgb(245, 239, 224);   /* ✅ 保持 */
  font-style: normal;          /* ✅ 不变 */
  font-weight: 400;            /* ✅ 不变 */
}

/* 段落分隔 */
.landmark-body p {
  margin-bottom: 0.75em;
}

/* 子标题 */
.landmark-body h3 {
  margin-top: 1.5em;
  font-size: 17px;
  font-weight: 600;
  letter-spacing: 0.02em;
  color: var(--gold-accent);
}

/* 标题保留 serif 做层级对比 */
.landmark-title {
  font-family: "Playfair Display", "Noto Serif SC", Georgia, serif;
  font-size: 28px;
  font-weight: 700;
}
```

配合 §6.4 方案 B 的渲染层段落自动分段函数，P00000 基本闭环。

**预期修复后的 UX 表现**：
- 拴日石 297 字从 1 段 → 2-3 段
- 读完时间：从"硬啃 2 swipe"→ "滑 1 swipe 读完第 1 段，再看需不需要继续"
- 字体从 serif → sans，无"斜体错觉"
- 字距归零，中文不再被人为拉开

## 12. 局限与警示

- **Zhihu 深度中文排版文档被拦**：本研究靠数英 + 七猫阅读器 + 搜索聚合补齐，但中文"圣经级"排版文档（如 Bobby Tung 的 Medium 系列）没能抓到。结论方向正确但细节可能有未覆盖项。
- **盘古之白没深度展开**：只给了原理和方案选择，没有具体基准测试。实际接入 pangu.js 后需要 QA 一轮所有中英混排字段。
- **15.5px vs 16px 的选择**：本研究推荐 15.5，但这是基于"比 17 要小但不要太小"的中间值。travel-web 实际应 **A/B 测 UX 感觉**，不是死信 15.5。
- **方案 B（渲染层兜底）有潜在副作用**：机器按"。"分段不一定符合作者原意（比如故意堆砌的长句在紧凑压迫感处）。建议 Dev 加一个 "数据层已有 `\n\n` 则尊重原意，否则兜底分段" 的短路逻辑。
- **本研究没覆盖字重**（"加粗"在 CJK 的正确用法）：UX 实测字重 400 OK，所以此维度没挖。但未来 landmark 子标题、emphasis 用法需要单独研究。

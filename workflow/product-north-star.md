# Product North Star · 产品锚点

**所有角色（PM / Dev / UX / QA / Demand / Explorer）做判断时必读**。
有分歧时，以本文件为准——不是以 PRD。PRD 是实现，本文件是方向。

**最后更新**：2026-04-22（CEO 第二次定性补强，Explorer 转录）

---

## 一句话定性

**给大陆+海外华人历史爱好者的跨场景历史路书 App**：旅行前种草 + 旅行中看懂 + （未来）旅行后召回。

---

## 产品形态定性（2026-04-22 硬线）

travel-web 是一款**"旅行+历史"垂直领域的独立应用（indie App）**，不是以下任何一种：

- ❌ 媒体品牌 / 内容品牌
- ❌ Newsletter / 订阅专栏
- ❌ 策展人 IP 项目
- ❌ KOL 个人品牌延伸
- ❌ 知识付费社群

**是**：
- ✅ 一款 indie App（类比：Bear 笔记 / Reeder / Things 3 / Fantastical）
- ✅ "旅行+历史"方向给自己和同好用的好工具
- ✅ CEO 角色 = **产品负责人 + 品味守门员**，不是作者 / KOL / 策展人
- ✅ 成功标准 = **用户真的在旅行前中场景用起来**，不是品牌影响力 / 内容阅读量

**含义**：
- 护城河 = 产品体验 + 数据库 + 垂直定位（可规模化、可传承）
- 内容生产靠流水线（demand + PM + CEO 过稿），不靠 CEO 亲自写每城
- 不做持续内容经营（Newsletter / 专栏 / 课程 / 社群）
- CEO 的 KOL 能力用于**产品上线时的首发推广**，不是日常运营

---

## 用户

**谁**：大陆 + 海外华人
**画像**：旅行 + 历史爱好者（不是只想拍照的游客，是到一个地方想搞懂"这地方为什么长这样"的人）
**原型用户**：CEO 本人

**关键信号**（以后识别"这是不是我们的用户"用这些）：
- 看到一座建筑会想知道"几几年建的"、"谁建的"、"为什么在这儿建"
- 不满足于"网红打卡点"，要能讲出故事的东西
- 会为"历史背景做足了功课"而自豪
- 对"含金量"敏感——虚的/编的/水的内容会让他反感

---

## 核心价值

> **旅行前、中、后的城市历史 + 景点历史 + 帮助旅行者更好体会景点历史的路书。**
> ——CEO 原话，2026-04-21

关键词拆解：
- **路书**：组织过的（不是零散 tip）、带动线的（有顺序）、行动导向的（告诉你该做什么）
- **历史**：含金量、有故事、有来源、不是水的网红文案
- **跨三段**：前中后，但现阶段不是三段齐飞（见下）

---

## 三段场景当前投入

| 阶段 | 现状 | 现阶段优先级 |
|---|---|---|
| 旅行**前**（种草 + 研究 + 决策） | 产品主要形态 | **重点** |
| 旅行**中**（现场体会历史） | 薄弱，但定位为重点 | **重点** |
| 旅行**后**（召回记忆） | 空白 | 未来（不在本阶段） |

**未来旅行后的形态**：让游客能回忆到当初的记忆（CEO 原话）。需要用户身份 + 个人存储，绕不开登录 / 小程序。

## 技术形态路线（2026-04-22 锁定）

| 阶段 | 形态 | 状态 | 优先级 |
|---|---|---|---|
| **阶段 1** | H5 静态站（现在）| 进行中（v3.5-exp 打磨中） | 当前重点 |
| **阶段 2** | H5 v1 上线（买域名 + CDN 迁国内 + 微信兼容）| 未启动 | v3.6 后 |
| **阶段 3** | **微信小程序 v1**（Dev-MiniApp 基于 H5 定稿 UI 重写） | 未启动 · 基础设施已预留（`miniprogram/` 目录 + 角色 + 数据共享约定） | 华人主战场，高优先级 |
| **阶段 4** | 简化 PWA manifest（给海外华人备用 "添加到主屏幕"）| 未启动 | 阶段 3 后补 |
| **阶段 5** | 原生 App（iOS/Android） | 未评估 | **不做**，除非阶段 3 微信小程序验证后用户量大到非做不可 |

**关键决策（2026-04-22）**：
- 华人主战场 = **微信小程序**，不是 PWA（微信 WebView 不支持 Service Worker）
- 全 PWA 不做，只做**简化 PWA**（manifest + 图标，不做 Service Worker）给海外华人用
- **小程序不着急启动**——H5 UI 定型后再开工，避免小程序 UI 做完要重写

## 已放弃 / 挂起的决策（2026-04-22）

- **α 方案（纯滚动 UI）**：2026-04-22 放弃，β（3 tab）胜出
- **Newsletter / 订阅专栏**：CEO 2026-04-22 明确不做——"不想夸夸其谈"
- **策展人 / 个人 IP 延伸**：CEO 2026-04-22 明确不做——"我不想当策展人"
- **商业化**：2026-04-22 CEO 表态"不着急、也没计划"——不在当前战略考虑范围
- 详见 `workflow/parked-decisions.md`

## 判断金线（所有角色决策时自问）

做任何改动 / 提任何需求前，问三件事：

1. **服务哪段场景？** 前 / 中 / 后。答不上来 = 可能在做错的事
2. **服务到"历史爱好者"画像了吗？** 还是在服务"一般游客"？两者需求不一样
3. **含金量够吗？** 凑数内容、水文案、占位符会让原型用户反感

## 这个文件什么时候该改

- CEO 主动调整定性
- 发现用户群漂移（真实用户和"大陆+海外华人历史爱好者"不符）
- 某段场景的优先级变化（比如启动小程序版 → 旅行后升为重点）

改之前先跟 CEO 确认。Explorer / PM 不得擅自修定性，可以提议。

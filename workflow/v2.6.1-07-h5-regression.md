## 三、Bug 详情

### Bug A · P0 · 城市页返回按钮在复杂动线中回不到主页

**复现**：
1. 打开主页
2. 点任意城市（如京都）→ 进入城市详情
3. 点景点卡片 → 进入 landmark 详情
4. 点 landmark 左上角 `←` → 返回到城市详情（这步 OK，因为 landmark 用的是 `navTo('#/city/${cityId}')`）
5. 点城市左上角 `←` → **期望：回主页。实际：跳回刚才那个 landmark 页**

**根因**：
- `js/app.js:329` 城市详情页返回按钮：`onclick="history.back()"`
- `history.back()` 只退一步，而第 4 步的 `navTo` 又 pushState 了一条新记录到 history 栈
- 所以"返回"按钮退到的是 **刚才离开的 landmark**，形成死循环

**证据**：jsdom 模拟显示，T2 动线结束后 hash = `#/city/kyoto/landmark/0`，三城市三端全部如此。

**CEO 截图吻合度**：京都 Hero 点返回无效/跳错，**完全匹配**此 bug——如果 CEO 在测试前随便点过一个景点，再回城市页按返回，就会陷入这个循环。

---

### Bug B · P0 · 冷启动分享链接直达城市页，返回按钮无效 / 跳出站

**复现**：
1. 从微信/X/直接粘贴 URL 打开 `https://.../index-h5.html#/city/kyoto`
2. 点左上角 `←`
3. **期望**：回主页。**实际**：无任何响应（`history.length === 1`，history.back() 无处可退），或在某些浏览器上直接关闭 tab / 退出站外

**根因**：同样是 `js/app.js:329` 的 `history.back()`。冷启动时 history 栈只有一条（当前页），back 无出路。

**影响范围大**：
- 分享场景（朋友圈 / 小红书 / 微博贴链接）→ 所有用户默认都是这个路径
- 书签 / 历史记录重新打开
- 刷新页面后的返回

**代码层**：
```js
// js/app.js:329
<button class="detail-back" onclick="history.back()">←</button>
// 同款还出现在：
// line 421 renderNotFound（城市未找到页）
// 有 2 处，两处都是同一个 bug
```

### Bug C · P1 · 修复路径已经存在但用户不知道

**事实**：
- `.more-cities-link` 在城市页中间（line 343），`onclick="navTo('#/')"` → **测试 PASS**

这也解释了 post-mortem 为什么错——**测试者可能测了 bottom-nav + more-cities-link 这两个"新加的出口"，没回头测老的左上角 `←`**。典型"修补时只盯着新功能，忘了回归老功能"。

## 四、代码层修复 brief（给 Dev-H5）

### 必改（2 行）

**`js/app.js:329`**（城市详情页返回按钮）
```js
// 错：
<button class="detail-back" onclick="history.back()">←</button>
// 改：
<button class="detail-back" onclick="navTo('#/')">←</button>
```

**`js/app.js:421`**（renderNotFound 未找到城市页返回按钮）
```js
// 错：
<button class="detail-back" onclick="history.back()">←</button>
// 改：
<button class="detail-back" onclick="navTo('#/')">←</button>
```

### 建议保留 `history.back()` 的地方（可选）

**`js/app.js:477` 和 `js/app.js:530`**（landmark 详情页返回按钮）目前是 `onclick="navTo('#/city/${cityId}')"`——这个写法**已经是对的**，因为 landmark 的"返回"语义就是回到当前城市。**不要改**。

### 为什么直接设成 `navTo('#/')` 而不是判断 referrer？

1. **语义一致性**：左上角 `←` 在本站 = "回主页发现"。这是 IA 层面的约定，不随用户动线变。
2. **取消歧义**：用户不需要理解"这个箭头回哪里"——永远回发现页。
3. **冷启动/分享场景自然修复**：history.back 在栈空时无效；navTo 不依赖栈。
4. **代码极简**：不需要做 `document.referrer` / `history.length` 的 if 判断。

如果 Dev-H5 觉得 back 按钮应该"根据来源智能返回"，需要 PM 先在 PRD 里定义好 IA 层的返回语义（建议改成"左上角恒定回发现页，次级返回用面包屑/底 tab"），否则不要加智能逻辑。

## 五、QA 流程反省（写给自己和 PMO）

### v2.6 上一轮为什么把 H-05 标成"修好了"

看 post-mortem 第 17 行：
> | 3. 城市内返回链不合理 | H-05 · bottom-nav + 更多城市入口 | ✅ | ✅ | 修好了 |

**判断错误的原因**：
1. QA 上一轮只测了 H-05 **新增**的两个入口（bottom-nav 和 more-cities-link），没回归测老的 `.detail-back` 左上角按钮
2. 测试设计者假设 "加了两个新出口 = 用户有办法回主页"，但**忽略了用户习惯**——用户第一反应永远是戳左上角
3. 没测试多层动线（T2）和冷启动（T3）；只测了最简单的 T1 路径
4. `.detail-back` 的 `onclick="history.back()"` 从 v2.5 以来就没动过，没人怀疑

### 下次怎么测（规则写入 QA playbook）

**每个"返回/导航"按钮必须测 4 种场景**，缺一即 FAIL：

1. **单步路径**：A → B → back → ?（最简）
2. **多层路径**：A → B → C → back → B → back → ?（测 history 栈）
3. **冷启动直达**：直接 URL 打开 B → back → ?（测 history.length=1）
4. **Referrer 外部**：从外部站跳入 B → back → ?（测 `document.referrer` 为空或跨域）

**通用原则**：
- `history.back()` **永远不是靠谱的站内导航**——它依赖外部状态（history 栈），而栈的深度是运行时的
- 站内导航应该显式声明目标（`navTo('#/')` / `navTo('#/city/xxx')`），不要靠浏览器历史栈猜
- 规则写入 `CLAUDE.md` 的 Dev-H5 section：禁用 `history.back()` 作为主要返回路径；可选择作为 fallback（"有 referrer 就 back，否则 navTo 根路由"）

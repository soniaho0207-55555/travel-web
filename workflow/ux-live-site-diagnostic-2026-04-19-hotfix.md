## 🎁 2 项未预期的大加分 · 超出 UX 诉求的升级

### 加分 A · 时间轴节点「展开阅读」承接 micro-story（方案 C 实现）

UX 上轮 Part 3 推荐方案 B（景点卡可展开 micro-story）。Dev 采用了**方案 C**——把 micro-story 全部迁到**历史时间轴节点的"展开阅读"折叠**里，分为三栏子结构：

- **背景**（如：日本 710 年迁都平城京之后 80 年间佛教寺院坐大 / 东大寺兴福寺僧兵左右朝政 / 道镜差点称天皇 / 桓武 781 年想跑路）
- **人物**（如：桓武天皇第 50 代、母亲百济人、动机一半政治一半族裔）
- **此时全球**（如：同年查理曼罗马加冕 / 阿拉伯阿拔斯智慧宫翻译希腊手稿 / 玛雅古典期）

**research-ux 判断**：方案 C 不是方案 B 的替代品，是更好的设计——把"帝国级 micro-story"放回它该在的层级（时间轴是帝国线程、景点卡是地标钩子）。上轮 UX 删掉的印加祭司 / 郭守敬 / 伽利略 / 起重机这些 micro-story 大概率会陆续以这种"展开阅读"方式回到时间轴节点。**UX 要核实 micro-story 是否真的被系统性保留**。

### 加分 B · 京都 Fushimi 景点的"全景 + 细节"图片叙事对

京都 Hero 图用了伏见稻荷的 torii 走廊全景（千本鸟居），重要景点 tab 里的"伏见稻荷大社"景点卡用了**狐狸守护雕像 + 鸟居背景的细节视角**。这是两张图有意识的对位——一张远景（"这个地方长什么样"）一张近景（"这里有什么值得看"）。

**对应镜头**：🟢「图片排序叙事」—— research-ux 判断这是该镜头的**教科书级落地**。travel-web 如果把这种"全景+细节"叙事对推广到所有景点，会直接拉起整站的图片品牌感。

# 🚨 本轮 P0 · 新发现必须 UX 判断

## P0-1 · 时间轴展开段**单段 246-270 字**触发弃读警报

**现场证据**：京都 794 平安京迁都节点，点开"展开阅读"后："背景"段实测 **246 字 / 315px 高度 / 38.7% viewport**。另一节点的"背景"段 270 字。

**对应镜头**：🟢「单段 viewport 上限 150 字」—— 150 字警报，250 字物理弃读

**UX 必答**：
1. 这 246-270 字的"背景"段读下去是否会像上轮拴日石 297 字那样想退出？
2. 如果"会想退"——方向是"按句号再分段"还是"内容太堆砌该 trim"？
3. 如果"不会想退"——因为是"展开后的深度内容"，用户心理预期不同？那"弃读警报镜头"在"主动展开后"是不是该放宽到 250 字？

这是本轮最重要的 taste 判断，**只有用户视角能答**。

展示要求：
- 京都 794 节点截图（展开后）
- 背景段滑到底需要多少次 swipe 的录屏
- 一段 100-150 字的主观判断

## P0-2 · **墨西哥城大教堂数据残缺**（CEO 亲自抓出）

**CEO 截图现场证据**：
- 点击墨西哥城大教堂 → 门票 tab → 只有「开放时间 08:00—20:00」，**没有价格/预订/交通信息**
- 点击 TIPS tab → 只有「该景点 Tips 补齐中」一行占位文字

**研究验证**：用 Node.js 扫描 `js/data.js` 全站 54 个景点：

```
总景点数：54
tips 完整：53    tips 空白：1（墨西哥城大教堂）
ticket 完整：53  ticket 仅 hours：0   ticket 空对象：1（墨西哥城大教堂）

残缺清单（1 条）：
  mexico-city > 墨西哥城大教堂 | tips: 0 | ticket: [] 
  （其他 53 个景点 ticket 都有 price+channels+bookingWindow+timingTip 四件套 + tips 4-6 条）
```

**结论**：这是 **PM 数据层漏网** — Dev 不能补（Dev 没有具体票价 / 预订渠道 / 交通细节的来源），必须 PM 去按"特奥蒂瓦坎"模板补齐：
```js
ticket: {
  price: '...',
  channels: [{name, url, note}, ...],
  bookingWindow: { peak, shoulder, offpeak },
  timingTip: '...'
},
tips: [
  { category: 'ticket' | 'timing' | 'walking' | 'photo' | 'cold' | 'food' | 'money', text: '...' },
  ...（建议 4-6 条）
]
```

**展示要求（UX）**：大教堂门票 tab 截图 + 对比同城"特奥蒂瓦坎"或"大神庙"完整 ticket tab 的截图

## P0-3 · **吴哥窟主殿图过暗**（CEO 亲自抓出）

**CEO 截图现场证据**：吴哥窟重要景点页，"吴哥窟主殿"卡片的封面图几乎全黑 — 水面倒影 + 五塔尖都是剪影，天空仅一抹粉色。

**研究分析** — **双重暗因**：

1. **图片本身是剪影风格**：URL 是 `upload.wikimedia.org/.../Sunrise_at_Angkor_Wat_Cambodia.jpg`。这是 Wikipedia Commons 的"吴哥窟日出"—— 作者原意就是逆光剪影艺术照片。对大图全屏观看是美的，对 160px 高卡片缩略图**信息量为零**。
2. **CSS overlay 加深**：`css/styles.css:1399` 给 `.landmark-card-simple .landmark-cover::after` 加了一层 `linear-gradient(to top, rgba(12,24,16,0.35) 0%, rgba(12,24,16,0) 55%)` — 卡片底部 35% 渐变暗遮罩（为了文案区对比度）。本身合理，但撞上剪影图 = 雪上加霜。

**修复路径**（三选一或组合，给 PM/Dev 参考）：

- **方案 A**（最小改动 - PM）：换 `wikiImage` 字段为白天清晰的正立面图。Wikipedia Commons 上候选：
  - `Angkor_Wat_temple.jpg`（正立面白天全景）
  - `Angkor_Wat_front_side.jpg`（西门正面）
- **方案 B**（学京都 Fushimi 叙事对 - PM+Dev）：Hero 用日出剪影（适合全屏），景点卡用白天全景（适合缩略图）。这是 UX 上轮点赞的「全景+细节叙事对」镜头的应用
- **方案 C**（降 overlay - Dev）：降 `rgba(..., 0.35)` 到 `0.2`。效果有限（图本身就暗），**不推荐单独用**

**推荐 A 或 B**。研究镜头「第一张图选择」命中红灯 — 当前是"地标全景明信片（剪影版）"+"廉价感的底色覆盖"。

**展示要求（UX）**：吴哥窟页面全部图片的叙事对照（Hero vs 景点卡），判断整站类似问题的景点还有哪些（其他日落 / 夜景 / 逆光图也可能踩坑）

## P0-4 · 伏见稻荷 era 字段错误："奈良·711年"

**现场证据**：京都 > 重要景点 > 伏见稻荷大社卡，era meta 显示 `奈良·711年`

**问题**：伏见稻荷大社**在京都伏见区**，不是奈良。711 年是**奈良时代前夜**（奈良时代是 710-794）——Dev 可能把"**时代名**"和"**地理位置**"混淆了：正确应该是「**奈良时代·711 年**」或「**平安前·711 年**」。

UX 展示要求：该卡片的截图 + 用红圈标出错误位置

修复路径（给 PM）：`js/data.js` 景点 era 字段检查——**同类错误可能在其他景点也存在**，建议全量扫一遍时代/地域字段

# 🔧 P1 · research-ux 技术测出，UX 只需确认

## P1-1（复用上版编号）· CDN 迁移仍未启动

**现场证据**：所有 18 张图片 host 仍是 `upload.wikimedia.org`。还有 Wikipedia API 跨墙调用（`en.wikipedia.org/w/api.php`）+ Google Fonts（`fonts.gstatic.com`）。

**影响**：大陆用户每次加载仍赌运气。京都 ORB 修了但 CDN 迁移没做 = 其他城市图可能随时掉线（Wikimedia 某些区域可达性波动）。

**状态**：这是 PMO / CEO 级决策，不是 Dev 能单方推进。B.1-B.4 研究弹药已就绪（腾讯 COS / 火山 ImageX 都给了具体方案）。

UX 展示要求：Network 面板 host 列截图 + Slow 3G 录屏证明加载体验

这决定了方案 C 的实现完整度。
